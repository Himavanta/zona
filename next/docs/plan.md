# Zona Next 实施计划

> 本文档是 Next 版本开发的唯一执行依据。`next/docs/` 下的其他文档为参考背景。语言规范以 `next/docs/spec.md` 为准。

## 语言设计锁定

Next 相对 V1 的语言变更，详见 `next/docs/spec.md`。核心变更：

1. **栈效应声明（强制）** — `@ 名称 输入:输出 body ;`，`:` 做签名分隔符
2. **数字类型分离** — `35` 整数(i)，`35.0` 浮点(d)，类型不匹配编译错误
3. **指针真实地址** — `:alloc` `:free` 返回真实指针，`:peek` `:poke` 替代 `&` `#`
4. **默认内联短词** — body ≤ 8 词元且非递归自动内联
5. **`.` 仅做成员访问** — 打印用 `:print` 原语
6. **`&` `#` 释放** — 预留给局部变量等未来特性

符号变更：

| 变更 | V1 | V2 |
|------|----|----|
| 签名分隔 | 无 | `:` |
| `.` 语义 | 打印 / 成员访问 | 仅成员访问 |
| 打印 | `.` | `:print` |
| 内存读 | `&` | `:peek` 原语 |
| 内存写 | `#` | `:poke` 原语 |
| `&` `#` | 占用 | 释放（预留局部变量） |

### 不做的事（现阶段）

- 不加局部变量语法 — 先用纯栈式写，验证语言设计
- 不加结构体 — 后续阶段
- 不加模块系统 — 后续阶段
- 不加错误处理 — 后续阶段
- 不加并发 — 后续阶段
- 不加字嵌套定义 — Forth 传统平级定义，不引入闭包复杂度

### 阶段 1：最小解释器

**目标**：能运行 fib，验证栈效应声明 + 类型分离的核心价值。

**产出**：`next/src/zona.c` — Next 解释器

**实现内容**：

1. **tokenizer** — 基于 V1 的 `zona.h`，新增：
   - 浮点字面量识别（含小数点的数字）
   - 栈效应声明的解析 `输入:输出`（`:` 前后的类型字母序列）

2. **栈模型** — 待决（见待决 6）。候选方案：
   - 分离栈（三栈模型：整数/浮点/指针各自独立）
   - 统一栈 + tagged value（运行时标签分派）
   - 统一栈 + 编译期类型追踪（推荐，无运行时标签开销）

3. **栈效应验证** — 编译字定义时，模拟执行 body，验证净栈变化与声明一致

4. **类型化执行** — 根据栈效应声明中的类型，生成类型化指令

5. **控制流** — `? ~ $` 与 V1 语义一致，`!` 做 else 分支标记

**必须通过的测试**：

```
_ 整数算术
3 5 + :print          _ 输出 8

_ 浮点算术
3.0 5.0 + :print      _ 输出 8.0

_ 类型错误
3 5.0 +               _ 编译错误：类型不匹配

_ 栈效应验证
@ add1 i:i 1 + ;
3 add1 :print         _ 输出 4

@ bad i:i :drop ;     _ 编译错误：声明产出1个，实际产出0个

_ 空签名
@ hello : 'Hello' :type 10 :emit ;
hello                  _ 输出 Hello

_ fib
@ fib d:d :dup 2.0 < ? $ :dup 1.0 - fib :swap 2.0 - fib + ;
35.0 fib :print       _ 输出 9227465
```

**不做**：字符串、内存操作、文件 I/O、模块、FFI、REPL

### 阶段 2：字节码 VM

**目标**：将 token 流解释升级为字节码执行，大幅提升性能。

**产出**：在 `next/src/zona.c` 中新增字节码编译器和 VM

**实现内容**：

1. **字节码指令集定义** — 类型化操作码：

   ```
   _ 栈操作
   OP_PUSH_I    (1 + 8 bytes)   _ 压入整数
   OP_PUSH_D    (1 + 8 bytes)   _ 压入浮点
   OP_DUP_I     (1 byte)        _ 复制栈顶整数
   OP_DUP_D     (1 byte)        _ 复制栈顶浮点
   OP_DROP_I    (1 byte)
   OP_DROP_D    (1 byte)
   OP_SWAP_I    (1 byte)
   OP_SWAP_D    (1 byte)

   _ 算术
   OP_ADD_I     (1 byte)        _ 整数加
   OP_ADD_D     (1 byte)        _ 浮点加
   OP_SUB_I     (1 byte)
   OP_SUB_D     (1 byte)
   OP_MUL_I     (1 byte)
   OP_MUL_D     (1 byte)
   OP_DIV_I     (1 byte)
   OP_DIV_D     (1 byte)
   OP_MOD_I     (1 byte)

   _ 比较（返回整数）
   OP_LT_I      (1 byte)
   OP_LT_D      (1 byte)
   OP_GT_I      (1 byte)
   OP_GT_D      (1 byte)
   OP_EQ_I      (1 byte)
   OP_EQ_D      (1 byte)

   _ 控制流
   OP_JZ        (1 + 4 bytes)   _ 条件跳转（整数 == 0 则跳）
   OP_JMP       (1 + 4 bytes)   _ 无条件跳转
   OP_RET       (1 byte)

   _ 调用
   OP_CALL       (1 + 4 bytes)  _ 调用字（索引）

   _ 打印
   OP_PRINT_I   (1 byte)
   OP_PRINT_D   (1 byte)
   ```

2. **字节码编译器** — 遍历 token 流，输出字节码序列。每个字编译为独立的字节码函数体。

3. **直接线程 VM** — 用 computed goto 分派字节码

4. **内联** — 短词（≤ 8 token，非递归）的字节码直接展开到调用点

**验证**：
- 阶段 1 的所有测试继续通过
- fib(35) 性能基准 — 预期从 V1 的 1.63s 降至 0.3-0.5s

## 待决事项

在写代码之前需要确定的设计问题。实现按阶段走，但语法设计应尽早定下来，避免解释器的数据结构白写。

### 待决 1：模块系统 ✅ 已决定

**决策**：全部导出，不区分公开/私有，命名空间即封装。

设计细节：

1. **导入语法** — `:use 名称 '路径'`，名称手动定义，不自动推导：
   ```
   :use math './math.zona'
   :use http '../lib/http.zona'
   ```

2. **成员访问** — `名称.词名` 作为单一 token 解析（tokenizer 层合并）：
   ```
   math.abs
   http.server.start
   ```

3. **全部导出** — 模块内所有词默认对外可见，没有 `:export`、`:hide`、`@+`/`@-` 等机制。命名空间本身就是边界，开发者自行控制命名约定（如内部词加 `internal-` 前缀）。

4. **允许嵌套深度** — `a.b.c.fn` 是合法的。每一层 `.` 是逐层取成员，模块和结构体统一语义。内部依赖自动可访问（和 Python 一致）。

5. **别名机制** — 通过定义新字实现重新导出或缩短路径：
   ```
   _ all.zona（barrel 文件）
   :use math './math.zona'
   :use io './io.zona'

   @ abs d:d math.abs ;       _ 别名
   @ neg d:d math.neg ;       _ 别名
   @ type pi: io.type ;       _ 别名
   ```

6. **别名与嵌套深度配合** — 嵌套太深时（如 `http.server.handler.middleware.auth.check`），用别名缩短：
   ```
   @ check i:i http.server.handler.middleware.auth.check ;
   ```

**边界情况分析**：

- **内部依赖变成公开** — `a` 模块 `:use b`，使用者可写 `a.b.fn`。这是可接受的，命名空间已提供隔离，开发者通过命名约定（`internal-` 前缀等）表达意图。和 Python 的 `module._private` 哲学一致。

- **不同模块的同名依赖** — `a` 和 `c` 都 `:use b` 但指向不同文件。通过 `a.b.fn` 和 `c.b.fn` 天然区分，不冲突。

- **循环引用** — `a` 引用 `b`，`b` 引用 `a`。V1 的去重机制保证每个文件只加载一次，不会无限循环。解释器需要处理"模块尚未加载完"的查找情况。

- **嵌套深度无限延伸** — 理论上 `a.b.a.b.a.b.fn` 可行，但实践中不会这么写。即使写了，去重机制保证不会真的无限递归。别名机制可以缩短过深的路径。

**对解释器的影响**：
- 字典结构从全局字典变为 `模块名 → { 词名 → Word }` 的嵌套字典
- tokenizer 需要识别 `xxx.yyy.zzz` 为单一 token（T_MEMBER 类型）
- `:use` 语义从"摊平文件"变为"导入模块命名空间"
- 成员访问在运行时逐层查找模块表

**影响**：解释器的字典结构（全局字典 vs 嵌套命名空间）取决于这个设计。如果先实现全局字典，后来加模块时需要大幅重构。

### 待决 2：结构体语法

需要在 `spec.md` 中定义以下语法细节：

### 待决 6：栈的物理模型 — 分离栈 vs 统一栈

阶段 1 计划中写入了三栈模型（整数栈、浮点栈、指针栈分离），但此决定未经过正式讨论确认。需要决定运行时栈的物理结构。

#### 方案 A：分离栈

```c
int64_t istack[1024];  // 整数栈
double  fstack[1024];  // 浮点栈
void*   pstack[256];   // 指针栈
int isp, fsp, psp;
```

优点：
- 类型检查免费 — 整数操作只碰 istack，浮点操作只碰 fstack，不需要运行时标签
- 缓存友好 — 同类型值连续存储
- 编译器直接映射到寄存器族 — istack → x 寄存器，fstack → d 寄存器

缺点：
- **跨类型栈操作复杂** — `:swap` 交换 `3 5.0`（一个在 istack 一个在 fstack）怎么办？必须根据栈效应签名知道操作数类型才能正确交换
- `id:d` 签名的字 — 消费一个整数一个浮点，`:swap` 需要跨栈搬移
- 字符串压入 `p i` 两个值跨了两个栈，`:drop` 需要知道 drop 哪个栈
- 调试时看三个栈比看一个栈更难理解
- `:swap` 等原语不再是简单的"交换栈顶两个值"，必须变成类型化的操作

#### 方案 B：统一栈 + tagged value

```c
typedef struct {
    uint8_t tag;  // TAG_INT, TAG_FLOAT, TAG_PTR
    union {
        int64_t ival;
        double  fval;
        void   *pval;
    };
} Value;

Value stack[2048];
int sp;
```

优点：
- `:swap` `:dup` `:drop` 等原语实现简单 — 不需要知道值的类型
- 调试直观 — 一个栈，从头看到尾
- 语义和 Forth 传统一致 — 栈是无类型的

缺点：
- 运行时标签检查开销 — 每次操作要读 tag 分派
- 每个值多占 8-16 字节（标签 + 填充），缓存不友好
- 编译器需要额外的去标签化优化才能达到寄存器传参的效果

#### 方案 C：统一栈 + 编译期类型追踪（推荐）

```c
double stack[2048];  // 统一栈，物理上存 double/int64_t/pointer（都是 64 位）
int sp;
```

物理上统一栈（每个槽位 64 位），但**编译器/字节码生成器根据栈效应签名知道每个位置的类型**，生成类型化的指令。

- `3 5 +` → `PUSH_I 3, PUSH_I 5, ADD_I` — 操作整数槽
- `3.0 5.0 +` → `PUSH_D 3.0, PUSH_D 5.0, ADD_D` — 操作浮点槽
- `:swap` 在编译时根据栈效应签名生成正确的字节码 — 如果栈顶两个是 `dd`，生成 `SWAP_D`；如果是 `ii`，生成 `SWAP_I`；如果是 `id`，生成 `SWAP_ID`（跨类型交换）
- 运行时无标签开销 — 字节码指令已经编码了类型信息
- `:dup` `:drop` 等原语不需要区分类型 — 64 位值的复制/丢弃是通用操作

优点：
- `:swap` 等原语在语义层仍然是"交换栈顶两个值"，和 Forth 传统一致
- 运行时无标签开销 — 类型信息在字节码指令中，不在值中
- 调试直观 — 一个栈
- 编译器可以根据类型信息生成最优代码（整数走 x 寄存器，浮点走 d 寄存器）
- 和 V1 虚栈方案一脉相承 — 编译期追踪类型，运行时零开销

缺点：
- 编译器需要精确的栈类型追踪 — 控制流汇合点（`?` 的两个分支）需要栈类型一致，否则编译错误
- 统一 64 位槽位浪费空间 — 整数只需要 32 位时也占 64 位（但 ARM64 上 64 位是自然字长，不算真正浪费）

**关键问题：控制流汇合点的栈类型一致性**

```
@ foo d:d
  :dup 0.0 < ?
    1.0 + !         _ 真：d d → d，栈顶是 d
    2 +              _ 假：d i → 类型冲突！
  ;                   _ 编译错误：两个分支栈类型不一致
```

编译器需要在 `?` 的两个分支汇合时验证栈类型一致。这是一个**约束**，但也是一个**特性** — 它防止了类型混乱的 bug。

**对解释器和编译器的影响**：
- 解释器：统一栈，字节码指令编码类型信息，VM 不需要运行时标签
- 编译器：统一栈，ARM64 代码生成时根据类型选择寄存器族
- 两者共享同一套栈类型追踪逻辑

**倾向**：方案 C — 统一栈 + 编译期类型追踪。在性能和简洁性之间取得最佳平衡。

### 待决 7：实现与设计的并行策略

核心问题：**先写最小解释器验证性能假设，还是先在 spec.md 中把模块/结构体语法定下来再写解释器？**

两种路径：

- **路径 A：先写解释器** — 快速验证栈效应 + 类型分离的性能价值。模块/结构体后面加时可能要重构字典和 tokenizer。
- **路径 B：先定语法再写解释器** — 把模块和结构体的语法写进 spec.md，解释器的数据结构一开始就为这些特性预留。但可能过度设计，在性能假设未验证时投入工程化特性。

建议：**路径 B 的折中** — 在 spec.md 中先定义模块和结构体的语法（只是文档，不写代码），解释器实现时数据结构预留扩展点，但不提前实现。

# Zona Next 实施计划

> 本文档是 Next 版本开发的唯一执行依据。语言规范以 `next/docs/spec.md` 为准，性能分析参考 `next/docs/perf-analysis.md`。

## 语言设计锁定

核心变更（详见 spec.md）：

1. **栈效应声明（强制）** — `@ 名称 输入:输出 body ;`
2. **统一类型词汇表** — `b h w l s d p p*`，栈效应用 `l d p` 子集
3. **数字类型分离** — `35` 整数(l)，`35.0` 浮点(d)，类型不匹配编译错误
4. **栈物理模型** — 统一 64 位栈 + 编译期类型追踪，`?` 分支汇合栈类型必须一致
5. **指针真实地址** — `:alloc` `:free` 返回真实指针，`:peek` `:poke` 替代 `&` `#`
6. **默认内联短词** — body ≤ 8 词元且非递归自动内联
7. **`.` 仅做成员访问** — 打印用 `:print` 原语
8. **`&` `#` 释放** — 预留给局部变量等未来特性

### 不做的事（现阶段）

- 不加局部变量语法 — 先用纯栈式写，验证语言设计
- 不加结构体 — 语法暂定在 spec.md 中，实现推迟到 FFI 阶段
- 不加错误处理 — 后续阶段
- 不加并发 — 后续阶段
- 不加字嵌套定义 — Forth 传统平级定义，不引入闭包复杂度

### 阶段 1：最小解释器 ✅

**目标**：能运行 fib，验证栈效应声明 + 类型分离的核心价值。

**产出**：`next/src/zona.c`

**已完成**：
- tokenizer（浮点/整数区分、栈效应签名解析、`:` 双重语义）
- 统一 64 位栈 + 运行时类型追踪
- 编译期栈效应验证（分支汇合类型检查、递归自引用）
- 类型化算术/比较操作
- 控制流 `? ! $ ~`

**验证通过**：
- `3 5 + :print` → 8
- `3.0 5.0 + :print` → 8
- `3 5.0 +` → 编译错误：类型不匹配
- `@ add1 l:l 1 + ; 3 add1 :print` → 4
- `@ bad l:l :drop ;` → 编译错误：栈效应不匹配
- `@ abs d:d :dup 0.0 < ? neg ! ;` → 正确
- `fib(35)` → 9227465

### 阶段 2：字节码 VM

**目标**：token 流解释升级为字节码执行，大幅提升性能。

**产出**：在 `next/src/zona.c` 中新增字节码编译器和 VM

**实现内容**：

1. **字节码指令集** — 类型化操作码（详见下方定义）
2. **字节码编译器** — 遍历 token 流，输出字节码序列
3. **直接线程 VM** — 用 computed goto 分派字节码
4. **内联** — 短词（≤ 8 token，非递归）字节码直接展开到调用点

字节码指令集：

```
_ 栈操作
OP_PUSH_L    (1 + 8 bytes)   _ 压入长整数
OP_PUSH_D    (1 + 8 bytes)   _ 压入浮点
OP_PUSH_P    (1 + 8 bytes)   _ 压入指针
OP_DUP       (1 byte)        _ 复制栈顶（64位通用）
OP_DROP      (1 byte)        _ 丢弃栈顶（64位通用）
OP_SWAP_LL   (1 byte)        _ 交换两个长整数
OP_SWAP_DD   (1 byte)        _ 交换两个浮点
OP_SWAP_LD   (1 byte)        _ 交换长整数和浮点

_ 算术
OP_ADD_L / OP_ADD_D
OP_SUB_L / OP_SUB_D
OP_MUL_L / OP_MUL_D
OP_DIV_L / OP_DIV_D
OP_MOD_L

_ 比较（返回长整数）
OP_LT_L / OP_LT_D
OP_GT_L / OP_GT_D
OP_EQ_L / OP_EQ_D

_ 控制流
OP_JZ        (1 + 4 bytes)   _ 条件跳转
OP_JMP       (1 + 4 bytes)   _ 无条件跳转
OP_RET       (1 byte)

_ 调用
OP_CALL       (1 + 4 bytes)  _ 调用字（索引）

_ 打印
OP_PRINT_L / OP_PRINT_D
```

**验证**：
- 阶段 1 的所有测试继续通过
- fib(35) 性能基准 — 预期从 V1 的 1.63s 降至 0.3-0.5s

### 阶段 6：ARM64 编译器 + FFI

**目标**：字节码 → ARM64 汇编翻译器，原生性能 + C 生态对接。

**产出**：`next/src/zonac.c`

**实现内容**：

1. **字节码 → ARM64 指令映射** — 一对一翻译
2. **C 调用约定** — 按 AAPCS64，`d:d` 的字编译为 `double func(double n)`
3. **函数序言/尾声** — callee-saved 寄存器保存/恢复
4. **汇编输出** — 生成 `.s` 文件，用 `cc` 链接
5. **FFI** — `:bind` 按 AAPCS64 生成 C 函数调用，类型使用 QBE 类型词汇 + `p*`

**验证**：
- fib(35) 性能 — 预期接近 C（1.0-1.3x）
- 阶段 3 的所有测试编译后通过
- FFI 调用 C 标准库函数成功

## 已决定的设计问题

### 统一类型词汇表

使用一套类型词汇表，不同上下文使用不同子集（详见 spec.md 3.1）：

```
完整词汇表：b  h  w  l  s  d  p  p*
大小：      1  2  4  8  4  8  8  8 字节
```

- **栈效应声明**：只用 `l` `d` `p`（栈是 64 位槽）
- **`:struct`**：用 `b`-`p`（精确内存布局）
- **`:bind`**：用 `w`-`p*` + `void`（FFI 边界精度 + 字符串指针特殊处理）

`p*` 仅用于 `:bind` 标注 `char*`，触发自动双向字符串转换，栈效应中展开为 `pl`。类型词汇直接对应 QBE IL，无中间映射层。

### 栈物理模型

统一 64 位栈 + 编译期类型追踪。运行时无类型标签，编译器根据栈效应签名生成类型化指令。`?` 分支汇合栈类型必须一致，`$` 豁免，`~` 处栈类型与字入口一致（详见 spec.md 3.6-3.7）。

### 模块系统

- 导入：`:use 名称 '路径'`，名称手动定义
- 成员访问：`名称.词名` 作为单一 token（T_MEMBER）
- 全部导出，命名空间即封装，无 `:export` 机制
- 别名：通过定义新字实现
- 字典结构：`模块名 → { 词名 → Word }` 嵌套字典

### 结构体语法 🔮 远期暂定

语法暂定在 spec.md 第 9 节，实现推迟到 FFI 阶段之后：
- `:struct Point x d y d` — 顶层声明，QBE 类型
- 自动生成构造器/析构器/字段读写
- 字段访问与模块成员访问统一为 `T_MEMBER`

### 资源上限

资源上限是解释器的实现细节，不是语言的语义约束。解释器用静态数组，编译后程序用 OS 资源限制。spec.md 不规定资源上限。

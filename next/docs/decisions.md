## 决策 1：字符串表示

**背景**：V1 字符串为 `(addr len)` 双值压栈。V2 有了指针类型 `p`，存在两种选择。

| 方案 | 栈效应 | 描述 |
|------|--------|------|
| A. 保持双值 `p l` | `( -- p l )` | 地址和长度分别压栈，V1 兼容，Forth 传统 |
| B. 单指针 | `( -- p )` | 字符串是单个指针，指向 `{ptr, len}` 结构体 |

**性能分析**：
- 方案 A：每个字符串占 2 个栈槽，`+` 传递长度值，`:type` 消耗 2 个值。栈操作多一倍。
- 方案 B：栈上只占 1 个槽，`:type` 只需 pop 一次。但需要隐式的字符串结构体和堆分配。

**性能倾向**：方案 B（少占栈槽，少一次 pop/push），但需要评估堆分配开销。

**✅ 已决定：方案 B — 单指针**。

**解释器实现**：B2 静态预分配，字符串头部 8 字节存长度，指针指向数据区。`'hello'` → `push_p(strings[i].data)`，`:type` 通过 `*(int64_t*)(ptr-8)` 读取长度。零运行时分配。

**编译器实现**：字符串放入 QBE/汇编 data section，单寄存器传参。FFI 调用 C 函数（如 `printf`）时天然一个 `x0` 传参，不需要拆成两个寄存器。

**`:type` 签名**：`p:`

## 决策 2：`:peek` / `:poke` 类型化方案

**背景**：V2 用 `:peek` `:poke` 替代 V1 的 `&` `#`。当前 spec 只定义了 `:peek p:l`（读 8 字节整数）。需要支持不同宽度的读写。

| 方案 | 描述 | 示例 |
|------|------|------|
| A. 扁平原语 | 每种宽度独立原语，类似 V1 | `:peek8 :peek16 :peek32 :peek64 :peekd :poke8 ...` |
| B. 上下文类型推断 | `:peek` 签名中写目标类型 | `:peek p:l`(8字节) `:peek p:w`(4字节) `:peek p:b`(1字节) |
| C. 运行时类型参数 | 栈上传入类型标记 | `p 'l' :peek` 或 `p :peekl` |

**性能分析**：
- 方案 A：编译时完全确定指令，零运行时分支。指令多但每条都是直线代码。
- 方案 B：需要 `w` `b` 出现在栈效应中（当前栈效应只有 `l d p`）。扩展栈效应类型词汇会增加编译器复杂度，但运行时同样是零开销。
- 方案 C：运行时判断类型，有分支开销。**性能最差**。

**性能倾向**：方案 A 或 B。方案 A 实现最简单且性能最优。方案 B 更"类型系统原生"但需要扩展栈效应词汇表。

## 决策 3：调试原语

| 原语 | V1 签名 | V2 建议签名 | 说明 |
|------|---------|------------|------|
| `:stack` | `( -- )` | `( -- )` | 打印栈内所有值，含类型标记 |
| `:trace` | `( flag -- )` | `( l -- )` | 非0开启单步追踪，0关闭 |

**是否纳入 V2 spec**：
- [ ] `:stack` — 纳入

**性能影响**：调试原语不影响正常执行性能（条件编译或运行时开关）。

## 决策 4：局部变量

**背景**：栈式代码可读性差，局部变量是缓解手段。`&` `#` 已在 V2 中释放，预留给局部变量。当前 spec 明确说"现阶段不加"。

| 方案 | 描述 |
|------|------|
| A. 永远不加 | 坚持纯栈式，依靠良好命名和短词提升可读性 |
| B. 远期加 | 保持 `&` `#` 预留，等核心功能稳定后再设计 |
| C. 字节码 VM 阶段预留指令 | 先在字节码指令集中加入 `OP_LOAD_VAR` / `OP_STORE_VAR`，但不暴露语法 |

**性能分析**：
- 局部变量本质是"命名栈槽"。在字节码/编译器层面可以映射到寄存器，**反而可能提升性能**（避免 `:swap` `:rot` `:over` 等栈重排操作）。
- 但如果实现为内存变量（栈帧中的 spill slot），则比栈操作慢。

**性能倾向**：方案 C — 指令集预留，但语法层面暂不加。等看到实际代码中栈重排成为瓶颈时再加语法。

## 决策 5：错误处理

**背景**：`( ) [ ] { }` 预留给错误处理。三个候选方案未决策。

| 方案 | 描述 | 符号需求 |
|------|------|---------|
| A. try-catch | 类似 C++/Rust 的异常捕获 | `( )` 或 `[ ]` |
| B. 错误码返回 | 字返回额外的错误标记，类似 Go | 不需要新符号 |
| C. result 类型 | 成功/失败封装在结构体中 | 不需要新符号（需要结构体） |

**性能分析**：
- 方案 A：try-catch 的"快乐路径"零开销（编译器生成跳转表），但异常发生时 unwind 成本高。
- 方案 B：每个调用都要检查错误码，有运行时分支开销。但简单可预测。
- 方案 C：类似 B，错误检查开销在线。

**性能倾向**：方案 A 的快乐路径零开销最适合性能优先场景。但实现复杂度最高。

**符号分配**：
- 如果用 try-catch：`( )` 还是 `[ ]`？
- 另一个给闭包/代码块？
- `{ }` 预留给什么？

**实现阶段**：建议至少在模块系统（阶段 4）之后，结构体（阶段 7）之前。

## 决策 6：独立编译

**背景**：Untitled-1.md 提到"没有独立编译"是大型项目杀手。当前 plan.md 的阶段 4 只覆盖模块系统（命名空间），不覆盖独立编译。

| 方案 | 描述 |
|------|------|
| A. 先做单文件编译 | 阶段 6 只编译单个 `.zona` 文件，`:use` 的文件全部展开内联 |
| B. 分开编译+链接 | 每个模块编译为 `.o`，最后 `cc` 链接。需要接口文件 |
| C. 全程序优化 | 类似 Rust，所有源码一起编译，跨模块内联 |

**性能分析**：
- 方案 A：编译快，但无跨模块内联。`:use` 展开意味着修改一个文件全部重新编译。
- 方案 B：增量编译快，但跨模块内联受限。
- 方案 C：运行时性能最优（跨模块内联），但编译慢。

**性能倾向**：先做方案 A（阶段 6），方案 C 作为远期优化阶段。方案 B 适合大型项目但复杂度高，暂不优先。

## 决策 7：并发模型

**背景**：讨论中提到可通过 `:bind` + `pthread` 实现基础并发。语言层面的并发模型需要深入设计。

| 方案 | 描述 |
|------|------|
| A. 仅 FFI 并发 | 通过 `:bind` 调用 pthread，不做语言层面支持 |
| B. 轻量级并发原语 | 提供 `:spawn` `:channel` `:lock` 等原语 |
| C. async/await | 语言层面的协程/异步 |

**性能分析**：
- 方案 A：FFI 调用开销，但对于 CPU 密集型任务（Zona 的甜点），线程切换开销本身是主要成本。
- 方案 B/C：如果实现为绿色线程/M:N 调度，可能比 pthread 更轻量。

**范围**：是否纳入 V2？还是明确列为 V3？

## 决策 8：编译器单遍 vs 多遍

**背景**：Untitled-1.md §7.8 讨论了编译速度与性能的矛盾。`perf-analysis.md` 中提到了寄存器传参等优化。

| 方案 | 描述 |
|------|------|
| A. 单遍编译 | 阶段 6 做单遍 ARM64 代码生成，简单快速 |
| B. 多遍优化 | 阶段 6 就加入内联、寄存器分配等多遍优化 |

**性能分析**：
- 方案 A：编译快，生成的代码有性能瓶颈（栈式语义 → 内存读写）。`perf-analysis.md` 估计单遍编译的 fib(35) 约为 C 的 5-10x。
- 方案 B：如果编译器利用栈效应签名，知道每个字消费/产生几个值，就能生成寄存器传参的代码。`perf-analysis.md` 估计可达 C 的 1.0-1.3x。

**性能倾向**：方案 B 更符合"性能优先"的目标。但工作量差异大——方案 A 是"字节码→ARM64 一对一翻译"，方案 B 需要做函数级别的寄存器分配。

**建议**：阶段 6 拆分为 6a（单遍，快速可用）和 6b（多遍优化，追求性能）。

## 决策 9：开发阶段顺序调整

**当前 plan.md 顺序**：
```
阶段 1: 最小解释器 ✅
阶段 2: 字节码 VM
阶段 3: 核心原语（字符串、I/O、文件、REPL）
阶段 4: 模块系统
阶段 5: 标准库
阶段 6: ARM64 编译器 + FFI
```

**建议调整**：将 `:stack` 调试原语提前到阶段 2，`:trace` 在阶段 2 一并实现。原因：
- 字节码 VM 开发时没有调试工具，定位 bug 会非常痛苦
- `:stack` 实现简单（遍历栈数组+打印类型），但收益巨大
- 不影响正常执行性能

**调整后建议**：
```
阶段 2: 字节码 VM + :stack + :trace
阶段 3: 核心原语（字符串、I/O、文件、REPL）
...
```

## `~` 循环的问题分析（已解决，本文档作记录备查用）

我通过大量测试定位到了根本原因。问题涉及 QBE 编译器后端和 V2 编译器架构的限制。

### 根因：QBE SSA 与虚栈的冲突

编译器用**虚栈（`vstack[]` + `vtype[]`）**追踪 QBE 临时变量。每个临时变量名（`%t0`, `%t1`...）在 QBE IR 中只能赋值**一次**（SSA 约束）。

```
@ decr-to-zero l:l :dup 0 = ? $ 1 - ~ ;
```

生成的 IR 大致为：
```
function l $zona_decr-to-zero(l %p0) {
@Lentry
    %t0 =l copy %p0          ← 输入值
    ...
@Lbody
    :dup → %t1 =l copy %t0
    0 =  → %t2 = ceql %t1, 0
    ? $  → jnz %t2, @ret, @cont
@cont
    1 -  → %t3 =l sub %t0, 1  ← 减1得到新值
    ~    → vsync() 推 %t3 到运行时栈
            jmp @Lbody         ← 跳回
```

**第二次进入 `@Lbody` 时**：
- 从运行时栈弹出新值 → `%t4 =l dtosi ...`
- 但 body 中 `:dup` 已硬编码引用 `%t0`（原始输入），不是 `%t4`
- `%t3 =l sub %t0, 1` 仍然用 `%t0` 计算，导致无限循环

### 为什么递归能工作

```
@ sumTo l:l :dup 0 = ? $ :dup 1 - sumTo + ;
```

递归调用 `sumTo` 会生成 `call $zona_sumTo`，QBE 每次调用都分配全新的栈帧，参数传入 `%p0`，完全独立——**没有 SSA 冲突**。

### 解决方案

有两个可行方向：

**方案 A（推荐，简单可靠）**：所有循环用**递归**表达。Forth 传统上就是用递归实现循环，fib(35) 已验证可达 C 性能。

**方案 B（需重构）**：`~` 的路由都走运行时栈。遇到 `~` 时，后续所有操作直接用 `emit_push`/`emit_pop`（读写运行时栈），不使用虚栈。但需要较大重构。

方案 A 更实际——当前标准库和测试可以全部用递归实现，无需修改编译器。要采用方案 B 吗？


# 主流的forth没有循环的能力吗，递归能彻底百分百代替循环吗

好问题。我来诚实地分析一下。

## Forth 有循环吗？

**有，而且很普遍。** 主流 Forth（Gforth、SwiftForth 等）都有完善的循环：

```
BEGIN ... UNTIL        ← 直到条件为真退出
BEGIN ... WHILE ... REPEAT  ← 带中间检查的循环
DO ... LOOP            ← 计数循环
```

`~` 本质上就是 `BEGIN ... AGAIN`（无限循环主体）+ 条件退出（`? $`）。

## 递归能否完全替代循环？

**理论**：可以。递归和图灵机等价，任何循环都能表达为递归。

**实际**：但有代价——每次递归生成一个 `call` 指令，有函数调用帧开销。不过对 Zona V2 来说：

- QBE 可能内联短递归函数
- fib(35) 已证明 0.04s = C 的 1.0x，**递归没有明显性能问题**
- 循环深度大时（如遍历 10 万次），递归可能导致栈溢出；循环则不会

## 我们真正的问题

**`~` 在 QBE SSA 形式下实现有根本限制**：QBE 是静态单赋值（SSA）IR，每个临时变量只能赋值一次。`~` 需要跳回代码块头部用新值重新执行，但 body 中的 QBE 指令已经固定引用了旧的 temp 编号。第二次进入 body 时，这些引用指向的还是旧值。

这不是"Forth 能不能有循环"的问题，而是 **"我们的编译器架构如何正确实现循环"** 的问题。当前 vstack 方案在设计上就没有考虑循环。

## 你的选择

1. **方案 A（当前方案）**：用递归表达循环，完全放弃 `~`。简单、可靠、已验证性能达 C 水平。但深度循环（10 万次+）可能栈溢出。

2. **方案 B（修复 `~`）**：将 `~` 翻译为**尾递归调用**——遇到 `~` 时生成 `call $zona_xxx(当前参数) ; ret`。这样：
   - 利用 QBE 的调用约定，每次都进全新栈帧
   - 不会有 SSA 冲突
   - 但每次迭代都有 `call` 开销（QBE 可能优化）
   - 仍然有栈深度问题

3. **方案 C（B 的改进）**：将 `~` 真正编译为**机器级分支**。但这需要放弃 vstack 纯用运行时栈，或对 QBE IR 做底层分支操作。工作量大。

# 详细解释一下c方案

## 方案 C 详解：将 `~` 编译为尾递归调用

核心思路很简单——**`~` 不是生成 `jmp @Lbody`，而是生成 `call $zona_xxx(当前参数) + ret`**。

### 当前问题

```
@ decr l:l :dup 0 = ? $ 1 - ~ ;
```

当前生成的 IR（会产生 SSA 冲突）：
```
function l $zona_decr(l %p0) {
@Lentry
    %t0 =l copy %p0
@Lbody                    ← 第一次进入没问题
    %t1 =l ceql %t0, 0    ← :dup 硬编码引用 %t0
    jnz %t1, @ret, @cont
@cont
    %t2 =l sub %t0, 1     ← 也引用 %t0
    vsync(%t2)            ← 把 %t2 推入运行时栈
    jmp @Lbody            ← 第二次进入: %t0 还是原始值！
                          ← %t2 在新迭代中没被使用
```

**SSA 的本质问题**：`%t0` 在 `@Lentry` 中定义一次。`@Lbody` 中的代码固定引用 `%t0`，第二次 `jmp @Lbody` 时 `%t0` 不会自动变成新值。

### 尾递归方案

`~` 生成的是**对自身函数的调用**：
```
function l $zona_decr(l %p0) {      ← 每轮迭代有全新的 %p0
@Lentry
    %t0 =l copy %p0
@Lbody
    %t1 =w ceql %t0, 0
    jnz %t1, @ret, @cont

@cont
    %t2 =l sub %t0, 1               ← 计算下一轮的值
    %t3 =l call $zona_decr(l %t2)   ← ~ → 尾递归调用! 全新的 %p0 = %t2
    ret %t3                          ← 传递返回值

@ret
    ret %t0                          ← $ 提前返回
}
```

**每轮 `call` 都是全新的 QBE 函数调用**，`%p0` 是全新的参数，由 QBE 分配新的寄存器——**完全没有 SSA 冲突**。

### 相比当前方案

| 方面 | 当前 `jmp @Lbody` | 方案 C: 尾递归 call |
|------|-------------------|---------------------|
| SSA 兼容 | ❌ 冲突 | ✅ 完美兼容 |
| 实现改动 | 已有 | 修改 `~` 处理约 15 行 |
| 性能 | 理论上更快（无 call 开销） | 每次迭代有 call/ret |
| 栈深度 | 无栈增长 | O(n)，深度循环会栈溢出 |
| QBE 能否 TCO | — | QBE 可能优化尾调用为 jmp |

### 性能 vs 栈深度权衡

**关键问题**：尾递归的栈深度是否可接受？

- fib(35) = 递归深度 35 × 大量调用 → **0.04s = C 的 1.0x** ✅
- 但 `decr 10000000` 会用 1000 万层栈 → ❌ 崩溃

**实际影响**：普通场景（树遍历、数学计算、解析）深度通常 < 1000，完全没问题。只有"裸计数到 N"这类场景才会爆栈。

### 是否值得做？

**如果要保留 `~` 的语义，这是最干净的方案**。相比纯运行时栈方案（所有 body 操作硬编码 push/pop），尾递归方案：

1. **改动量小**：只改 `~` 处理逻辑，不影响 `gen_token` 其他分支
2. **没有模式切换复杂度**：不需要"循环模式 vstack vs 运行时模式"
3. **QBE IR 干净**：每个 body 操作还是直接生成 `add`/`sub` 等 QBE 指令，而不是 `call $zona_push/pop`
4. **类型安全**：`~` 处 vstack 类型必须匹配输入签名，编译器自然检查

### 实现方式（伪代码）

在 `gen_word` 的 `~` 处理中：
```
~  →  for i = n_in-1..0: args[i] = vpop(&type)
      emit "call $zona_xxx(type %t, ...)"
      if n_out > 0: emit "ret %t_ret"
      else: emit "ret"
      return 1  (结束函数，后面无代码)
```

## 总结

方案 C = **`~` = 尾递归调用**。用函数调用语义自然解决 SSA 冲突，改动量最小（~15 行），性能足够。唯一的代价是深度循环会消耗栈空间。

# 我理解不了这是什么意思，我的想法是我的～语法这不就是个goto吗，回到某个位置重新执行，什么都不用管，为什么会有这种问题

你说得对，直觉上 `~` 就是个 goto，回到开头重新执行。

## 问题出在哪——用最直白的方式解释

我们的编译器在编译 `@ decr l:l :dup 0 = ? $ 1 - ~ ;` 时，按顺序逐个 token **一次性生成** QBE 指令。生成的 IR 大致是：

```
; 编译阶段生成的代码（一次性，固定不变）
@Lbody:
    %t1 =l copy %t0       ← :dup (从 vstack 拿到 %t0)
    %t2 =l ceql %t1, 0    ← 0 =
    jnz %t2, @ret, @cont
@cont:
    %t3 =l sub %t0, 1     ← 1 -
    ... 把 %t3 存到运行时栈 ...
    jmp @Lbody             ← ~ 跳回！
```

**问题在这里**：当 `jmp @Lbody` 跳回去时，CPU 重新执行 `%t1 =l copy %t0`。但 `%t0` 是**第一次进入函数时的原始输入**，不会因为你跳回去了就自动变成当前值。

对比真实 CPU 汇编：
```asm
; 你期望的效果（每次执行 r0 都更新）
loop:
    sub r0, r0, #1       ; r0 每次减 1
    cmp r0, #0
    bne loop
```

但在 QBE 的 SSA 形式中，`%t0` 就像一块**刻了名字的石头**——它在 `@Lentry` 里被定义了，内容永远不会变。`%t3 =l sub %t0, 1` 永远是原始值减 1，不是"当前值减 1"。

**所以问题总结**：
- 我们的编译器生成 QBE IR 时，用的是**编译时的虚栈状态**——那时 `%t0` 在栈上，就写了 `%t0`
- 运行时跳回去，执行的还是这些写死的指令
- `%t0` 永远不会自己变成循环后的新值

## 三种修法

| 方案 | 本质 | 改动量 |
|------|------|--------|
| **C: 尾递归** | `~` = `call 自己(新参数) ; ret` — 每次都走函数调用，参数自动更新 | ~15行 |
| **纯运行时栈** | 含 `~` 的字所有操作走运行时栈（`push`/`pop`），不用虚栈 | ~50行重构 |
| **虚栈加 phi 桥接** | 在循环头从运行时栈 reload 值到新 temp | ~20行 |

# 虚栈加 phi 桥接 是什么意思

## φ (phi) 节点：SSA 处理循环的标准方式

先看一个普通的 C 循环：
```c
int decr(int n) {
    while (n != 0)
        n = n - 1;    // n 被"覆盖"了
    return n;
}
```

在 SSA 形式中，一个变量不能"覆盖"——`n` 的每个版本都得是新的名字。所以 LLVM/QBE 用 **phi 节点** 来解决。

### 对比理解

**CPU 汇编（你的直觉——就是 goto）：**
```asm
decr:
    # n 在 r0 中
loop:
    cmp r0, #0         # 检查 r0
    beq end
    sub r0, r0, #1     # r0 减 1（寄存器可变）
    b loop
end:
    ret
```

**QBE SSA（当前问题）：**
```
@Lentry
    %t0 =l copy %p0    ← n 的初始版本
@Lbody
    %t1 =w ceql %t0, 0 ← 总是用 %t0（原始 n），不是当前 n！
    jnz %t1, @ret, @cont
@cont
    %t2 =l sub %t0, 1  ← 也是用原始 n
    jmp @Lbody
```

**QBE SSA + phi 桥接（修复方案）：**
```
@Lentry
    %t0 =l copy %p0    ← n 的初始版本
    jmp @Lbody

@Lbody
    ; φ（读作"法艾"）= 选择器
    ; 如果从 @Lentry 来 → 用 %t0（初始值）
    ; 如果从 @cont 来   → 用 %t2（新值）
    %val =l phi @Lentry, %t0, @cont, %t2

    %t1 =w ceql %val, 0  ← 看！用的是 %val，它会自动变
    jnz %t1, @ret, @cont

@cont
    %t2 =l sub %val, 1   ← 减 1 得到新值 %t2
    jmp @Lbody           ← 跳回 @Lbody，phi 选 %t2 作为新 %val
                         ← 第三次跳回，phi 又选新的 %t2
                         ← 每次迭代 %val 都是最新值！

@ret
    ret %val
```

### phi 做了什么

`%val =l phi @Lentry, %t0, @cont, %t2`

翻译成人话：**"%val 的值取决于我从哪里来——如果是第一次进入函数，用 %t0；如果是循环回来的，用 %t2"**。

这就是 SSA 世界里"变量被更新"的机制。本质上 phi 就是一条"根据来路选值"的特殊指令。

### 对我们项目的意义

**当前 vstack 方案**：编译器在遍历 body 时，从 vstack 拿 temp 编号写指令。遇到 `~` 时，把这些 temp 编号直接写进了指令——但下次循环这些 temp 不变。

**phi 桥接方案**：在 `@Lbody` 开头为 vstack 上每个值插入 phi 节点。之后 body 所有指令都引用 phi 的结果，而不是原始 temp。这样循环就自然工作了。

**改动量很小**：在 `gen_word` 中，只需要在 `@Lbody` 标签后生成 phi 节点 -> 用 phi 结果填充 vstack -> body 继续用 vstack。`~` 处 vsync 后 jmp 回 `@Lbody`。

**QBE 完全支持 phi**，这是标准用法。要试试吗？

# 性能上的影响呢，会有什么利弊

## Phi 的性能影响

### 先说结论：Phi 是**零开销**的，比尾递归方案好得多

Phi 节点不是运行时指令——它是编译器内部的"接线员"。

### 简单类比

把 SSA 的值想象成**快递包裹上的地址标签**：

```
@Lbody
    %val =l phi @Lentry, %t0, @cont, %t2
    %t1 =w ceql %val, 0
```

Phi 做的事情在运行时 **什么都不做**。它只是在编译阶段告诉 QBE："嘿，第一次循环用 `%t0` 这个寄存器，之后用 `%t2` 那个寄存器。"

QBE 的寄存器分配器会直接把 `%t0` 和 `%t2` 分配到**同一个寄存器**里，phi 就消失了。运行时只有 `sub` 和 `jmp`。

### 对比三种方案

```
@ decr l:l :dup 0 = ? $ 1 - ~ ;
n=100000 次循环
```

| 方案 | 运行时代价 | 栈使用 | 说明 |
|------|-----------|--------|------|
| **Phi 桥接** | **一条 `sub` + 一条 `jmp`** | O(1) | 和 C 的 while 编译结果一样 |
| **尾递归 call** | 10 万次 `call` + 10 万次 `ret` | O(n) | 每次迭代压栈、弹栈 |
| **纯运行时栈** | 每次算术前 pop + 后 push，都走内存 | O(1) | 值不在寄存器里，全在 `$stack` |

### 数值对比（估算）

假设一次 `sub` = 1 单位时间：

| 操作 | 耗时 | 1 万次循环 |
|------|------|-----------|
| `sub` | 1 | 1 万 |
| `call` + `ret` | ~5-10 | 5-10 万 |
| 运行时 push/pop（2 次内存读写） | ~20-40 | 20-40 万 |

### Phi 的额外好处

**当前还有一个隐藏的性能浪费**：`~` 前面的 `1 -` 计算出 `%t3 = sub %t0, 1`，然后 `vsync()` 把它**写入运行时栈 `$stack`**。下次进入 `@Lbody` 又从运行时栈**读回来**。

```
%t3 =l sub %t0, 1      ← 在寄存器里算好了
stored %t3, $stack[...] ← 存到内存 (vsync)
... jmp @Lbody ...
loadd %t4, $stack[...]  ← 又从内存读回来
```

**Phi 方案完全消除这个浪费**——phi 节点直接让寄存器里的值"流动"到下一轮循环，不走内存。

### 总结

| | Phi 桥接 | 尾递归 | 纯运行时栈 |
|--|---------|--------|-----------|
| 运行时开销 | **0（编译期概念）** | 每次 call/ret | 持续 push/pop |
| 栈增长 | 无 | O(n) ❌ | 无 |
| 保持寄存器传参 | ✅ | ✅ | ❌ 全走内存 |
| vsync 浪费 | 消除 | 消除 | 需要 |
| QBE 支持 | 标准功能 | ✅ | ✅ |
| 实现难度 | 小 | 小 | 中 |

**Phi 在各方面都是最优方案**。而且它是 LLVM、GCC 等所有工业编译器处理循环的标准方式——不是变通技巧，而是正确的做法。要做吗？

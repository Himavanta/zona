# 内存读写设计探索

> 记录 `&` / `#` 语义的多次讨论和所有候选方案。
> 当前决定见 `decisions.md` 决策 2。

---

## 背景

Zona 需要从真实 C 指针读写内存。V1 用 `&`（读一个单元）和 `#`（写一个单元），操作的是内部 `mem[]` 数组的偏移量。V2 改为操作真实指针。

核心矛盾：内存读写有不同宽度（1/2/4/8 字节），有不同解释（整数/浮点）。如何在类型系统和语法层面表达。

---

## 方案 A：扁平原语（V1 风格）

每种宽度和类型独立命名：

```
:peek8  :peek16  :peek32  :peek64  :peekd
:poke8  :poke16  :poke32  :poke64  :poked
```

**优点**：编译时完全确定，零运行时分支。实现极简。

**缺点**：16 个原语，命名噪音大。V2 有栈效应签名，不需要靠名字区分类型。

**性能**：解释器零分支，编译器直接映射 QBE load/store 指令。

---

## 方案 B：签名重载（`:peek` / `:poke` 单条原语）

一条原语，签名区分类型：

```
:peek p:l    _ 读 8 字节整数
:peek p:d    _ 读 8 字节浮点
:poke lp:    _ 写 8 字节整数
:poke dp:    _ 写 8 字节浮点
```

**优点**：栈效应签名已经告诉了类型，不需要原语名重复信息。

**缺点**：窄宽度（1/2/4 字节）无法用栈效应表达——`l d p` 都是 8 字节栈槽。窄宽度访问需走结构体。

**性能**：解释器中 `:peek` 需要根据运行时类型追踪判断读 `l` 还是 `d`（一次类型检查）。编译器零开销。

---

## 方案 C：保留 `&` `#` 单符号

用最短语法表达最高频操作。签名区分类型：

```
&         _ 内存读 ( p -- l ) 或 ( p -- d )
value addr #   _ 内存写 ( value addr -- )
```

**符号代价**：`&` `#` 是两个宝贵的单符号，占用了就不能做局部变量。

**栈顺序**：`value addr #` 意味着 `#` 先弹 addr（栈顶）再弹 value（次栈顶）。构建地址时 addr 已经压在栈上，后算出的 value 自然在顶部，不需要 `:swap`。

**性能**：解释器和编译器都是零开销——编译时类型已知，生成对应指令。

---

## 方案 D：引用模型（类似字符串）

内存读写统一为引用类型，与字符串表示一致。内部 `{addr, value}` 或 `{addr, len}` 结构体，外部只暴露一个 `p`。

```
:cell  _ ( addr value -- ref )  创建单元引用
&      _ ( ref -- value )        读引用
#      _ ( value ref -- )        写引用

_ 示例：
100 :alloc 42 &:cell   _ 创建引用
& .                    _ 读 42
```

**优点**：
- 栈效应统一——所有引用都是一个 `p`，不暴露内部结构
- 与字符串模型一致：字符串是 `{len, data}`，内存单元是 `{addr, value}`
- 消除 `:swap`——不需要手动调整栈顶顺序
- 编译器阶段零开销——引用在栈上传，QBE 可以拆成两个寄存器传参

**缺点**：
- 解释器每次 `:cell` 需要堆分配 `{addr, value}` 结构体，高频场景有性能开销
- 解释器和编译器的行为不同——解释器真的分配，编译器优化掉
- 额外概念：用户需要理解"引用"是什么

---

### 方案 D 变体：不叫 `:cell`，直接改造 `#`

保持 `&` `#` 符号名，但改变语义——`#` 不再做写操作，而是创建一个引用（`value addr #` → 返回引用）。`&` 仍然是读。

```
100 :alloc 42 :swap #   _ value addr # → 返回 ref
& .                     _ ref → value
```

这样 `#` 和 `&` 就是引用的创建/读取，语义对称。

---

### 方案 D 变体：嵌入长度

不只是 `{addr, value}`，加上宽度信息——`{addr, value, width}`。这样同一个引用模型也能处理窄宽度：

```
p 42 8 :cell     _ 8 字节整数引用
p 0xff 1 :cell   _ 1 字节引用
```

`&` 根据引用的宽度自动选择 `loadub` / `loadsw` / `loadl` / `loadd`。

编译器阶段零开销——宽度是编译时常量。解释器阶段引用结构体里有 width 字段，运行时分支。

---

## 总结

| 方案 | 原语数量 | 语法长度 | 解释器开销 | 编译器开销 | 窄宽度 |
|------|:--------:|:--------:|:----------:|:----------:|:------:|
| A 扁平 | 16 | 6-7 字符 | 零分支 | 零开销 | ✅ |
| B 签名重载 | 2 | 5 字符 | 类型检查 | 零开销 | ❌ |
| C `&` `#` | 2 符号 | 1 字符 | 零开销 | 零开销 | ❌ |
| D 引用模型 | 2 符号 | 1 字符 | 堆分配 | 零开销 | ✅（含 width） |

**性能优先建议**：编译器阶段方案 C/D 都是零开销。方案 D 解释器有堆分配成本但编译器能消除。方案 B 解释器有类型检查分支，不推荐。

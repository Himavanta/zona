# Zona V2 语言重设计与编译器架构

> ⚠️ **历史文档**：本文档记录 V2 设计初期的探索和讨论。**大量内容与最终方案不同。**
> - 最终语言规范见 [`spec.md`](spec.md)
> - 实际编译器实现见 [`compiler.md`](compiler.md)
> - 废案汇总见 [`rejected.md`](rejected.md)
> - 性能分析见 [`perf-analysis.md`](perf-analysis.md)
>
> **关键分歧速览**：`( )` 括号签名 → V2 用 `d:d`；类型后缀 → V2 用小数点区分；ARM64 直出 → 最终用 QBE；`noinline`/`:export` 关键字 → 均被否决。各节顶部的 ⚠️ 标记具体说明差异。

> 本文档探讨从零重新设计 Zona 语言和编译器的方案，目标是在保持 Forth 极简精神的同时，从语义层面解决性能瓶颈，并支撑大型乃至超大型项目。

## 一、核心洞察

**栈是编译期概念，不是运行时数据结构。**

当前 Zona 把栈当成运行时的全局数组，但 Forth 的栈在语义上是一种匿名的数据流表达方式 — 不需要给中间值起名字，它们按顺序流动。源码层面写 `3 5 +`，表达的是"3 和 5 流入 +"。编译器应该把它编译成寄存器操作，而不是往一个全局数组里存三次。

**性能瓶颈不在后端，而在语义。**

之前的分析（见 `perf-analysis.md`）表明：当前 Zona 距 C 2.5x 的性能差距根源是三座大山 — 全局数组栈、统一 double 转换税、无函数签名。这些都是语言语义层面的决策，不是编译器实现或后端选择的问题。无论用 QBE IR 还是手写汇编，如果语义模型不变，生成的代码本质上一样。

**Forth 的极简哲学和大型项目的工程化需求是矛盾的，但可以找到平衡点。**

每加一个特性（模块、类型、结构体、错误处理），Zona 就离 Forth 远一步、离 C/Rust 近一步。但如果不加，它就永远只能做小程序。V2 的定位是：保留 Forth 栈式表达力，同时具备工程化可用性。

## 二、语言设计变更

### 2.1 栈效应成为强制类型系统

> ⚠️ 本节使用 `( d -- d )` 括号语法，最终方案简化为 `d:d`（见 spec.md §三）。括号保留给未来错误处理。

每个字**必须**声明栈效应，它就是类型签名：

```
@ fib ( d -- d ) :dup 2.0 < ? $ :dup 1.0 - fib :swap 2.0 - fib + ;
@ neg ( d -- d ) 0 :swap - ;
@ emit ( i -- ) ... ;
@ key ( -- i ) ... ;
```

`( d -- d )` 表示消费一个 double、产生一个 double。括号里的字母是类型标记（`i` = 整数, `d` = 浮点, `p` = 指针, `s` = 字符串）。

编译器在编译期**验证栈效应** — 如果一个字声明 `( d -- d )` 但内部栈不平衡，编译报错。这消除了 Forth 最恼人的运行时栈溢出/下溢 bug，同时给了编译器最关键的优化信息：**函数的参数数量、类型、返回值数量和类型**。

有了栈效应声明，编译器就能将参数通过寄存器传递，返回值通过寄存器返回 — 和 C 一模一样。

### 2.2 类型后缀，统一栈

> ⚠️ 本节提出的类型后缀（`35u8` 等）被否决。V2 最终方案：数字类型由小数点决定，`35` = 整数 l，`35.0` = 浮点 d。无后缀。

保留一个栈，但给值加类型标记：

```
35        _ 整数字面量，类型 w（32位整数）
35.0      _ 浮点字面量，类型 d（64位浮点）
35u8      _ 字节，类型 b
'hello'   _ 字符串（地址+长度）
```

编译器根据类型选择指令 — `35 3 +` 生成 `add w`，`35.0 3.0 +` 生成 `add d`。栈效应声明里的类型信息让编译器知道函数参数的寄存器分配：

```
@ fib ( d -- d ) :dup 2.0 < ? $ :dup 1.0 - fib :swap 2.0 - fib + ;
```

编译器看到 `( d -- d )`，就知道：参数走 d0 寄存器进来，返回值走 d0 寄存器出去。

**消除的转换税**：当前 Zona 的 `2 < ? $` 需要 `比较→整数→double→整数→跳转` 的疯狂转换链。有了类型区分，`2.0 <` 直接生成 `fcmp + b.mi`，和 C 一致。

### 2.3 默认内联

> ⚠️ 本节提出的 `noinline` 关键字被否决。内联是编译器优化策略，不侵入语法。内联本身为远期任务。

Forth 的短词传统使得内联特别有价值。新设计中：

```
@ neg ( d -- d ) 0 :swap - ;              _ 默认内联（body ≤ N token）
@ fib ( d -- d ) noinline ... ;           _ 显式不内联（递归等）
```

或者更简单 — 编译器根据 body 长度和是否递归自动决定。短词（≤ 8 token）默认内联，递归词和长词不内联。

### 2.4 指针是真实地址

当前 Zona 的 `&`/`#`（内存读写）通过全局 `mem[]` 数组 + 间接寻址函数，地址编码为 double。

新设计中，指针就是 `l` 类型的整数（真实 C 地址），不再编码为 double 偏移量：

```
:alloc ( i:size -- p:ptr ) ...   _ 返回真实 malloc 地址
:free ( p:ptr -- ) ...           _ 接收真实地址
@ peek-d ( p:ptr -- d:val ) ...  _ 直接 loadd
@ poke-d ( d:val p:ptr -- ) ...  _ 直接 stored
```

这让编译器能直接用 `loadd`/`stored`，不需要 `$zona_mem_at` 间接层。

### 2.5 结构体：数据封装

> ⚠️ 本节使用 `:struct ... :end` 块语法，最终方案定为更简洁的 `:struct Name type type ...` 行内声明。构造器/析构器自动生成也被否决（见 rejected.md §17）。```
_ 定义
:struct Task
  pid   i
  state i
  name  p
  next  p:Task
:end

_ 使用
Task.new ( -- p:Task )          _ 构造器：在堆上分配并初始化
task.pid ( p:Task -- i )        _ 字段读取
task.pid! ( i p:Task -- )       _ 字段写入
Task.free ( p:Task -- )         _ 析构器
```

字段访问用 `.` — `task.pid` 是结构体字段，`io.open` 是模块成员，语法一样但语义由编译器区分（结构体字段是偏移量加载，模块成员是函数调用）。

结构体也让 FFI 自然 — C 的 `struct` 可以直接映射。

### 2.6 模块系统：显式导出 + 独立命名空间

> ⚠️ 本节提出的 `:export` 显式导出被否决。V2 最终方案采用"全部导出，命名空间即封装"（见 rejected.md §14）。"`.` 仅用于成员访问"也被否决，V2 保留 `.` 的空格双语义。当前 V1 的 `:use` 把所有模块摊平到全局字典，在大型项目中必然导致命名冲突。

V2 方案：

```
_ io.zona
@ open ( p:path i:flags -- i:fd ) ... ;
@ close ( i:fd -- ) ... ;
@ read ( i:fd i:n -- p:buf i:nread ) ... ;

:export open close read
```

`@` 定义的词默认模块内部可见，`:export` 显式导出。外部 `:use` 时只看到导出的词。

调用方式：

```
:use io
io.open ( p:path i:flags -- i:fd )    _ 点号访问
io.close ( i:fd -- )
```

这解决了命名冲突 — 两个模块都有 `@ init`，但一个是 `net.init`，一个是 `fs.init`。

**点号的语义**：V2 中 `.` **仅用于模块/结构体成员访问**，打印用 `:print` 或其他原语。V1 中 `.` 有空格和无空格两种语义（打印 vs 成员访问），V2 统一为成员访问。

### 2.7 独立编译：接口文件

大型项目必须支持增量编译 — 不能改一行就全量重编译。

接口文件方案（编译时生成或手写）：

```
_ io.zona 接口文件
:module io
:export
  open   ( p:path i:flags -- i:fd )
  close  ( i:fd -- )
  read   ( i:fd i:n -- p:buf i:nread )
:end
```

调用方只需要读接口文件，不需要读实现。接口文件包含词名、栈效应、类型 — 编译器需要的一切信息。

编译管线变成：

```
io.zona → io.o（独立编译，包含接口元数据）
main.zona → main.o（只需要 io 的接口）
链接器 → 可执行文件
```

这是 C 的 `.h` / `.c` 模式，但用栈效应声明代替了头文件中的函数声明。

### 2.8 错误处理

当前 Zona 没有任何错误传播机制。大型项目中每个系统调用都可能失败，每一层都要处理。

方案一：沿用 C 的惯例，返回值表示错误

```
@ open ( p:path i:flags -- i:fd|i:err ) ... ;
_ 调用后手动检查 fd 是否为负数
```

方案二：在栈效应类型系统中引入 `i?`（可能错误的整数）

```
@ open ( p:path i:flags -- i?:fd ) ... ;
_ i? 表示"要么是 fd，要么是错误码"
_ 编译器可以强制调用方处理错误分支
```

这是 Rust `Result<T, E>` 的思路，但用 Forth 的方式表达。实现复杂度较高，但值得探索。

方案三：用保留的 `()` 做捕获块

```
( :fopen )   _ 执行 :fopen，如果失败将错误码压入错误栈而不是崩溃
( :fopen ) ? _ 检查是否有错误
```

目前暂不确定方案，需要更多设计思考。但大型项目必须有错误处理机制。

### 2.9 局部变量语法（可选）

Forth 的栈式表达在数据流简单时很优雅，在数据流复杂时很痛苦。`a b c f g h + - * swap rot +` 已经很难读。大型项目需要可读性。

```
_ 传统 Forth 风格
@ distance ( d:x1 d:y1 d:x2 d:y2 -- d:dist )
  :swap - :swap - * :swap    _ 这已经很难读了

_ 局部变量风格（V2 可选）
@ distance ( d:x1 d:y1 d:x2 d:y2 -- d:dist )
  |dx| x2 x1 -
  |dy| y2 y1 -
  dx dx * dy dy * + .sqrt
```

`|name|` 从栈上取值并绑定名字。这破坏了 Forth 的纯粹性，但在大规模代码中可能是必要的妥协。

### 2.10 保留符号的未来用途

```
( ... )    _ 可能用于错误处理（捕获块）
[ ... ]    _ 可能用于结构体/数组字面量
{ ... }    _ 可能用于闭包/代码块
```

## 三、编译器架构：直接生成 ARM64 汇编

> ⚠️ **本章方案已被否决。** V2 最终采用 QBE 后端，而非直接生成 ARM64。QBE 已验证能生成与 C 同等性能的代码（fib(35) 0.04s，C 的 1.0x）。本章内容保留作为设计探索的记录，详见 rejected.md §3 和 decisions.md §3。

### 3.1 为什么选择直接生成汇编

有了栈效应声明后，QBE 的主要价值（寄存器分配、指令调度）可以被一个针对 Zona 语义优化的极简代码生成器替代：

- Zona 的栈模型天然映射到寄存器 — 栈顶 N 个元素就在 d0-d7 / x0-x7
- 不需要通用寄存器分配器 — 调用约定和栈效应已经定义了值的流转
- 不需要指令调度 — ARM64 的乱序执行已经足够好
- 代码生成器可以内联 TOS 缓存策略，而不是依赖后端优化

ARM64（Apple Silicon）的指令集很规整，常用指令不到 100 条。

### 3.2 FFI 能力

**直接生成汇编不意味着失去 FFI 能力。** ARM64 的 C 调用约定（AAPCS64）是公开且稳定的规范，Zona 编译器可以自己实现：

#### AAPCS64 核心规则

| 项目 | 规则 |
|------|------|
| 整数参数 | x0-x7（前 8 个） |
| 浮点参数 | d0-d7（前 8 个） |
| 整数返回值 | x0 |
| 浮点返回值 | d0 |
| 溢出参数 | 从右到左压入栈 |
| Callee-saved | x19-x28, d8-d15 |
| 栈对齐 | 16 字节 |

实际上，FFI 在直接汇编方案下**可能更灵活**，因为 Zona 编译器完全控制调用约定的生成：

- QBE 方案：`:bind` 必须通过 QBE 的函数调用机制，受限于 QBE 支持的类型
- 汇编方案：`:bind` 直接按 AAPCS64 规则生成 `bl` + 参数搬移，任何 C 函数都能调用

#### Zona 自身调用约定 vs C 调用约定

关键设计决策：Zona 的字之间用什么调用约定？

**方案 A：Zona 字用 C 约定**

Zona 字编译为标准 C 函数。`( d -- d )` → 参数走 d0，返回走 d0。FFI 调用零成本转换。

优点：FFI 简单，调试友好（gdb/lldb 直接看符号）。
缺点：Zona 内部调用也受限于 C 约定的寄存器保存/恢复开销。

**方案 B：Zona 字用自定义约定，FFI 做 thunk 转换**

Zona 内部用更激进的约定（如 TOS 缓存 — 栈顶始终在 d0，不保存 callee-saved），调用 C 函数时生成 thunk 做约定转换。

优点：Zona 内部调用更快。
缺点：复杂度高，调试困难，thunk 有运行时开销。

**推荐方案 A** — 用 C 约定。理由：

1. C 约定在 ARM64 上已经很高效 — 前 8 个参数走寄存器，和自定义约定差异不大
2. FFI 零成本 — 不需要 thunk
3. 和系统库（libc, libm, raylib 等）直接互操作
4. 调试器、profiler 直接工作
5. 实现简单 — `bl $function_name` 就是函数调用

当 Zona 的 fib 编译为 `double zona_fib(double n)` 时，它的调用约定和 C 的 `double fib(double n)` 完全一致，性能也完全一致。

#### `:bind` 的新设计

当前 `:bind` 在汇编方案下的实现：

```
:bind myPuts 'puts' int char*
```

编译器解析 `:bind` 时记录 C 函数名和类型签名。在调用点：

1. 从 Zona 栈弹出参数（已在寄存器中）
2. 按 AAPCS64 放入正确的寄存器（char* → x0，int → x1，等等）
3. 生成 `bl _puts`
4. 将返回值压入 Zona 栈（放入 d0 或 x0）

这个过程和 QBE 方案做的本质上一样，只是我们自己控制寄存器搬移而不是让 QBE 做。而且因为我们完全控制代码生成，可以做 QBE 做不到的优化 — 比如连续 FFI 调用时复用寄存器中的参数。

### 3.3 代码生成策略

#### 寄存器分配

核心思想：**栈顶元素就在寄存器中**。

对于 `( d -- d )` 的字：
- 进入时：参数已在 d0（调用者按 C 约定放入）
- `:dup` → 保存 d0 到 d1（或临时变量）
- `+` → `fadd d0, d0, d1`
- 退出时：返回值在 d0

对于 `( d d -- d )` 的字：
- 进入时：第一个参数在 d0，第二个参数怎么办？

这里有个微妙的问题 — C 约定不支持"两个浮点参数都走 d0"。解决方案：

**栈顶元素在 d0，其余参数通过栈传递（真正的硬件栈，不是全局数组）**：

```
@ add-pair ( d d -- d )
    _ 进入时：d0 = 栈顶（第二个 d），[sp] = 栈次顶（第一个 d）
    ldr d1, [sp]        _ 从硬件栈读次顶
    fadd d0, d1, d0     _ 相加
    add sp, sp, #8      _ 弹出硬件栈
    _ 退出时：d0 = 结果
```

但这又引入了内存访问。更好的做法是：**多参数函数用多寄存器传参**：

```
@ add-pair ( d d -- d )
    _ 进入时：d0 = 第一个 d，d1 = 第二个 d（C 约定支持！）
    fadd d0, d0, d1
    _ 退出时：d0 = 结果
```

AAPCS64 支持前 8 个浮点参数走 d0-d7。所以 `( d d -- d )` 的字编译为 `function d $add_pair(d %a, d %b)`，完全合法。

**问题在于调用侧** — Zona 的栈是无名的，调用者怎么知道 d0 和 d1 分别对应栈上的哪个值？

答案：**编译器根据栈效应声明自动处理**。调用 `add-pair` 时，编译器知道要消费 2 个 double，就从虚栈/寄存器中取出两个值放入 d0 和 d1。

#### 栈溢出处理

当栈深度超过寄存器容量时（> 8 个浮点值或 > 8 个整数值），溢出到硬件栈：

```asm
; 栈顶 8 个值在 d0-d7，第 9 个值溢出到 [sp]
str d8, [sp, #-8]!     _ 压栈
ldr d8, [sp], #8       _ 弹栈
```

这在实践中极少发生 — 绝大多数 Zona 字的栈深度 < 4。

### 3.4 内联实现

内联在汇编层面就是直接把被调用字的指令序列插入到调用点，跳过 `bl`/`ret`：

```
@ neg ( d -- d ) 0 :swap - ;

_ 调用 neg 时，不生成 bl $zona_neg
_ 而是直接插入：
fmov d1, d0           _ :swap（d0 和 d1 交换角色）
fsub d0, d1, d0       _ 0 - n（实际是 0 - n → neg）
```

内联后 QBE 的非 SSA 修复能力不再可用，需要编译器自己处理临时变量。但因为 Zona 的内联词极短（≤ 8 token），临时变量冲突几乎不是问题。

### 3.5 编译管线

```
zona 源码
  ↓ tokenizer（复用 zona.h）
  ↓ 解析器（新增：栈效应解析、类型推断）
  ↓ 编译器前端（虚栈 + 栈效应验证）
  ↓ 代码选择器（Zona IR → ARM64 指令选择）
  ↓ 汇编输出（.s 文件）
  ↓ cc -o output（链接 libc/libm 等）
```

不再经过 QBE。直接输出 `.s` 文件，用系统的 `as` 或 `cc` 汇编和链接。

## 四、预期性能

### 理想目标：fib(35) 接近 C

当前 C fib(-O2)：0.043s
当前 Zona 编译器：0.106s

V2 Zona 编译为：

```
_zona_fib:
    fmov d1, #2.0
    fcmp d0, d1
    b.mi Lbase
    stp d9, d8, [sp, #-32]!
    stp x29, x30, [sp, #16]
    fmov d1, #-1.0
    fadd d1, d0, d1
    mov.16b v9, v0
    mov.16b v0, v1
    bl _zona_fib
    mov.16b v8, v0
    fmov d0, #-2.0
    fadd d0, d9, d0
    bl _zona_fib
    fadd d0, v8, d0
    ldp x29, x30, [sp, #16]
    ldp d9, d8, [sp], #32
Lbase:
    ret
```

这和 C 编译器输出的结构**完全一致**。差异可能来自：

- C 编译器的指令调度更优（V2 可以后期优化）
- C 编译器可能做尾调用优化（V2 可以实现）

预期 fib(35)：0.045-0.055s，约为 C 的 1.0-1.3x。

## 五、与当前 Zona 的兼容性

> ⚠️ 本节描述的独立源文件方案（`zona2.c`/`zonac2.c`）已被否决。V2 统一在 `next/src/zonac.c`。解释器已从项目中移除（见 rejected.md §1, §20）。

V2 是不兼容的重设计。主要破坏性变更：

| 特性 | V1 | V2 |
|------|----|----|
| 栈效应声明 | 无 | 强制 |
| 数字类型 | 统一 double | 整数/浮点分离 |
| 函数参数 | 全局栈传递 | 寄存器传递 |
| 字面量 `35` | double 35.0 | 整数 35 |
| 字面量 `35.0` | 同上 | 浮点 35.0 |
| 指针 | double 编码的偏移量 | 真实地址（l 类型） |
| 编译后端 | QBE | 直接 ARM64 汇编 |
| 内联 | 无 | 默认内联短词 |

### 迁移策略

不需要迁移。V1 和 V2 是两种语言，共存于同一仓库：

```
src/
  zona.h      _ V1 共享头文件
  zonac.c     _ V1 编译器（QBE）
next/
  src/
    zonac.c   _ V2 编译器（QBE）
```

V2 的标准库从零重写，但语义上尽量复用 V1 的设计。

## 六、开发优先级

1. **栈效应解析器** — 在 zona.h 中新增栈效应的解析和验证逻辑
2. **类型系统** — 整数/浮点字面量区分，类型传播
3. **ARM64 代码生成器** — 最小可用版本，只支持算术、比较、条件跳转、函数调用
4. **FFI 支持** — `:bind` 按 AAPCS64 生成调用代码
5. **内联** — 短词在调用点展开
6. **完整原语** — 逐步补齐内存、I/O、字符串等原语
7. **模块系统** — 显式导出、独立命名空间
8. **结构体** — 数据封装与字段访问
9. **独立编译** — 接口文件、对象文件、增量编译
10. **错误处理** — 错误传播与强制检查
11. **优化** — TOS 缓存、常量折叠、尾调用优化
12. **并发模型** — 远期目标，预留架构空间

## 七、开放问题

### 7.1 Apple Silicon 之外的平台

直接生成 ARM64 汇编意味着只支持 Apple Silicon（和 Linux ARM64）。如果未来需要支持 x86-64：

- 方案 A：写第二个 x86-64 后端（工作量约等于再写一个编译器）
- 方案 B：保留 QBE 作为 x86-64 后端，ARM64 走直接汇编
- 方案 C：只支持 ARM64（Apple Silicon 市场份额持续增长）

### 7.2 解释器

V2 的早期版本包含一个解释器（`zona.c`），后因维护成本翻倍且无性能优势被移除。V2 仅保留编译器模式。详见 rejected.md §1。

### 7.3 栈效应推断

已定结论：不自动推断，强制显式声明。栈效应声明（`d:d`）同时承担函数签名和类型标记的双重职责，是 V2 类型系统的核心设计。自动推断不可靠（Zona 允许通过 `:bind` 调用任意 C 函数，运行时行为无法静态推断），且会失去签名的文档价值。

理论上可以 — 编译器可以模拟执行每个字，追踪栈的增减。但 Zona 的控制流（`? ~ $`）使得推断不总是精确。建议：

- 编译器**可以推断**简单字的栈效应
- 复杂字（有 `?` 的字）**必须手动声明**
- 推断结果可以 `:check` 验证，也可以手动覆盖

### 7.4 字符串和结构体

字符串在 V2 中怎么处理？当前是 `( addr len )` 双值压栈。V2 有了指针类型后，字符串可以是单个指针值（指向栈上分配的结构体），或者保持双值传统。这需要更多设计思考。

### 7.5 栈式代码在大规模下的可读性

`3 5 +` 很清晰，但 `a b c f g h + - * swap rot +` 已经很难读。Forth 社区称之为"只写语言" — 写的时候知道在干什么，三个月后读不明白了。

Linux 级别的项目需要很多人读同一段代码。栈式表达在数据流简单时很优雅，在数据流复杂时很痛苦。

可能的缓解方案：
- 局部变量语法（见 2.9）— 破坏纯粹性但提升可读性
- 良好的词命名 — 短词 + 描述性命名本身就是文档
- 代码规范和约定 — 类似 C 的编码风格指南

### 7.6 工具链生态

Linux 依赖 git、make、clang、gdb、valgrind、perf、gcov 等完整工具链。Zona 当前只有一个 `:stack` 和一个 `run.sh`。大型项目需要：
- 调试器（断点、单步、栈查看）
- 性能分析器（函数级耗时统计）
- 测试框架（单元测试、集成测试）
- 包管理 / 依赖管理

### 7.7 并发模型

Linux 内核的核心挑战是并发 — 自旋锁、互斥锁、RCU、原子操作。Zona 当前没有并发故事。

架构上预留 — 结构体和模块系统为锁、通道等并发原语提供了封装基础。`:bind` 可以绑定 `pthread_create`、`pthread_mutex_lock` 等 C 函数。但语言层面的并发模型（async/await、actor、CSP 等）需要深入设计。

### 7.8 编译速度与全程序优化的矛盾

直接生成汇编的单遍编译器编译速度很快，但全程序优化（内联、跨模块优化）需要多遍。大型项目要么接受次优性能，要么接受慢编译。C 选择了前者（头文件模型），Rust 选择了后者（cargo 全量编译）。

V2 初期可以接受次优性能（单遍编译 + 适度内联），后期按需引入多遍优化。

## 八、超大规模支撑能力评估

### V2 能支撑多大规模？

| 规模 | 行数 | 示例 | V2 能否支撑 | 关键依赖 |
|------|------|------|-------------|----------|
| 小型 | < 1K | 脚本、工具 | 能 | V2 基础功能即可 |
| 中型 | 1K-10K | 游戏引擎、Web 服务 | 能 | 模块系统 + 结构体 |
| 大型 | 10K-100K | 小型 OS、数据库 | 勉强 | + 独立编译 + 错误处理 |
| 超大型 | 100K+ | Linux 级 | 困难 | + 工具链 + 并发 + 人才池 |

### 当前设计的五个大型项目杀手

1. **全局平坦命名空间** — 已通过模块系统（2.6）解决
2. **没有独立编译** — 已通过接口文件方案（2.7）规划，待实现
3. **没有数据抽象** — 已通过结构体方案（2.5）规划，待实现
4. **没有错误处理** — 已提出三个候选方案（2.8），待决策
5. **没有并发模型** — 远期目标，已预留架构空间

### 根本性局限

即使加上所有设计，Zona V2 做超大型项目仍有先天局限：

1. **栈式代码的可读性** — 数据流越复杂越难读，局部变量是缓解但不是根治
2. **工具链生态** — 需要从零建设调试器、profiler、测试框架
3. **人才池** — 会写 Forth 的人极少，协作规模受限于学习曲线
4. **Forth 极简哲学与工程化需求的内在张力** — 每加一个特性就离 Forth 远一步

### 定位

**Zona V2 不应该追求成为"能写 Linux 的 Forth"，而应该追求成为"保留 Forth 表达力的、工程化可用的语言"。**

它的生态位是：比 C 更安全（栈效应验证），比 Rust 更简单（没有生命周期），比 Go 更有趣（栈式表达），但能写出真正的大型软件。可支撑中大型项目（万行级，如小型 OS 或游戏引擎），超大型项目（千万行级）需要工具链和生态的长期建设。

# Zona 性能优化深度分析

> 本文档整理自关于 Zona 编译器性能优化、Forth 方言高性能实现、以及编译策略与硬件对齐的讨论。

## 一、当前优化状态

### 已完成的四个阶段

Zona 编译器 (`zonac.c`) 的核心优化思路是**将栈式机器的运行时开销逐步消除**，利用 QBE IR 的特性让 Zona 代码更接近寄存器机器：

#### 阶段一：虚拟栈骨架（fib(35) 0.48s → 0.29s）

编译时用 `vstack[]` 数组追踪 QBE 临时变量名，直线代码（算术、数字）不再生成 `call $zona_push / $zona_pop`，而是直接用 `copy`、`add` 等 QBE 指令操作临时变量。控制流边界（`? ~ $`）通过 `vsync()` 将虚栈刷回运行时栈。

#### 阶段二：栈原语虚栈化（0.29s → 0.26s）

`:dup :swap :drop :over :rot` 在虚栈上直接重排索引，零运行时指令。

#### 阶段三：运行时函数内联展开（0.26s → 0.16s）

`emit_push`/`emit_pop` 将 7 条 QBE 指令直接展开到调用点，消除 call/ret 开销。充分利用 QBE 的非 SSA 自动修复能力 — 展开产生大量同名临时变量，QBE 自动处理。

#### 阶段四：比较-跳转融合（边际收益）

在 `?` 处，若虚栈仅有一个条件值（`vsp==1`），`vpopr` 直接获取，生成 `ceqd → jnz` 融合代码。

### 累计效果

| 测试 | 优化前 | 优化后 | 总提升 |
|------|--------|--------|--------|
| fib(35) | 0.48s | 0.16s | **3x** |

### 全语言对比（fib(35)）

| 排名 | 语言 | 时间 | vs C |
|------|------|------|------|
| 1 | C (-O2) | 0.043s | 1x |
| 2 | Bun | 0.041s | 0.95x |

## 二、性能差距的根源：三座大山

### 栈式机器 vs 寄存器机器 — 本体论差异

Zona 的语义模型是**所有值在一个全局数组 `$stack[]` 中流转**。编译器无法摆脱这个约束，因为：

- 函数没有参数、没有返回值 — 参数通过全局栈传递
- 每个字调用前后的栈状态是隐式的，编译器不知道调用者传了几个值、被调用者消费/产生几个值
- 所以每次调用必须先 `vsync()` 把虚栈刷到内存，调用后再从内存读回来

对比 C 的 fib：参数通过 d0 寄存器传入，返回值通过 d0 寄存器返回。中间值保存在 v8/v9 寄存器中。**零内存访问**。

Zona 的 fib 每次递归调用至少需要 **6-8 次内存读写**（读 sp、算偏移、读 stack 数组、写 stack 数组、写 sp），而 C 的 fib 在热路径上**零内存访问**。

### 统一 double — 比较操作的双重转换税

Zona 生成的 `2 < ? $` 对应的汇编：

```asm
fcmp d0, d1          ; 比较，结果在条件标志
cset w1, mi          ; 标志→整数(0/1)
scvtf d0, w1         ; 整数→double     ← Zona 要求所有值是 double
fcvtzs w1, d0        ; double→整数     ← ? 需要整数给 jnz
cmp w1, #0           ; 与0比较
bne L109             ; 跳转
```

C 只需要：

```asm
fcmp d0, d1          ; 比较
b.mi LBB0_2          ; 直接跳转
```

Zona 因为统一 double 的设计，被迫做 `比较结果 → 整数 → double → 整数 → 与0比较 → 跳转` 的转换链。在 2900 万次递归调用中被反复执行。

### 没有函数签名 — 调用约定的灾难

Zona 的字 `@ fib ... ;` 编译为 `function $zona_fib()` — 零参数、零返回值：

- **调用者**：必须先把虚栈刷到内存数组，再 call
- **被调用者**：必须从全局内存数组读参数
- **返回后**：调用者再从全局内存数组读结果

对比 C/Hare：函数有类型签名，QBE 按平台 ABI 传参（arm64 上前 8 个浮点参数走 d0-d7 寄存器），全部零内存开销。

### 为什么语言简单 ≠ 编译容易

语言的简洁性是面向人的，编译器需要的是面向机器的信息。Zona 的语法极简，意味着编译器能从代码中提取的语义信息也极简：

| 信息 | C/Hare | Zona |
|------|--------|------|
| 参数类型 | `double n` | 无（从栈上取） |
| 参数个数 | 1 | 未知 |
| 返回值类型 | `double` | 无（压到栈上） |
| 变量类型 | 编译器推断 | 全是 double |
| 局部变量 | 寄存器分配 | 全局数组 |
| 栈效应 | 无 | 每个操作都要维护 |

Rust 的复杂语法给编译器提供了大量信息 — 生命周期、类型、所有权、借用检查。这些信息让 LLVM 能做激进优化。Hare 比 Rust 简单，但仍有类型系统和函数签名，让 QBE 能用平台 ABI 传参传返回值。

**Zona 的问题不是后端（QBE），而是前端给后端的信息太少。**

## 三、高性能 Forth 方言是怎么做的

### TOS 寄存器缓存

这是最关键的技术。传统 Forth 的数据栈在内存中，但高性能实现把**栈顶元素（Top-Of-Stack）始终保存在一个硬件寄存器中**：

```
传统 Forth:  每次 :dup → 从内存读栈顶 → 写到内存
TOS 缓存:    每次 :dup → 只是复制一个寄存器 → 零内存访问
```

在 x86/ARM 上，把数据栈顶固定映射到一个寄存器（如 x86 的 `eax`/ARM 的 `d0`）。有些实现甚至做**双 TOS 缓存**（栈顶两个元素都在寄存器中），对 `:swap`、`+` 等双操作数指令尤其有效。

### 子程序线程 + 内联 + 超级指令

传统 Forth 用**间接线程**（Indirect Threading）— 每个词是一串地址，运行时逐个跳转。每次跳转都是间接分支，在现代 CPU 上**完全击穿分支预测器**。

高性能 Forth 改用**子程序线程**（Subroutine Threading）：每个词编译成一串 `call`/`ret`。然后：

- **内联原语**：`:dup` `:drop` `+` 这些底层操作不是 `call`，而是直接插入机器码
- **超级指令**：把高频组合（如 `:dup +` → `fadd`）识别出来，合成单条机器码

做完这些后，生成的代码和 C 编译器的输出在结构上没有区别。

### 实际案例

**SwiftForth / VFX Forth**：商业 Forth，在 SPEC 基准上能达到 C 的 70-90%。核心策略是 TOS 缓存 + 子程序线程 + 内联。它们也有类型信息（至少知道浮点/整数），比较操作不需要 double→int→double 的往返。

**Chuck Moore 的极端路径**：Forth 发明者认为问题不在编译器，而在硬件。他设计了 Forth 专用芯片（F18A 等），指令集就是 Forth 原语，没有通用寄存器，栈就是寄存器，每个核极小，一个芯片上放几百个核。在这种硬件上 Forth 是原生语言。

## 四、绕过 IR 直接生成汇编，是否是正解

### 不是 IR 的问题

假设绕过 QBE 直接生成 ARM64 汇编，对于当前 Zona 的 fib，语义要求的操作不变：

```asm
; 无论用 QBE IR 还是手写汇编：
ldr w0, [_sp]              ; 读 sp
ldr d0, [_stack + ...]     ; 读数组
fcmp d0, d1                ; 比较
str d0, [_stack + ...]     ; 写数组
str w0, [_sp]              ; 写 sp
bl _zona_fib               ; 调用（参数在数组里，不在寄存器里）
```

全局数组读写是 Zona 语义规定的，不是 QBE 强加的。**生成汇编还是生成 IR，如果语义模型不变，生成的代码本质上一样。**

### 真正的问题：执行模型与硬件对齐

现代 CPU 的执行模型：
- 数据在**寄存器**中运算
- 函数参数通过**寄存器**传递（ARM64 的 d0-d7）
- 内存访问是**昂贵的**（几十个周期的延迟）

Zona 当前的执行模型：
- 数据在**全局数组**中
- 函数参数通过**全局数组**传递
- 寄存器只是临时中转

"面向硬件"的正解不是换代码生成方式，而是**改变编译策略，让栈操作映射到寄存器操作**。

### 用 QBE IR 也能做到

如果编译器知道 fib 消费 1 个值、产生 1 个值，生成的 IR 可以是：

```
function d $zona_fib(d %n) {     # 参数在寄存器！
@start
    %cmp =w cltd %n, d_2          # n < 2?
    jnz %cmp, @base, @recurse
@base
    ret %n                         # 返回值在寄存器！
@recurse
    %n1 =d sub %n, d_1             # n-1，全在寄存器
    %r1 =d call $zona_fib(d %n1)  # 参数通过寄存器传递
    %n2 =d sub %n, d_2
    %r2 =d call $zona_fib(d %n2)
    %r =d add %r1, %r2
    ret %r
}
```

这和 C 编译器生成的 IR 几乎一样。**不需要绕过 QBE — 需要的是改变编译器生成 IR 的方式。**

## 五、三条可能的路径

### 路径 A：保持当前语义，在编译器中做更激进的优化

- **栈效应推断**：分析每个字消费/产生几个值
- **TOS 缓存**：约定栈顶元素始终在某个寄存器
- **自定义调用约定**：用寄存器传参

Forth 世界主流高性能实现走的路。可以在 QBE 之上实现，不需要换后端。但编译器复杂度大幅上升，推断不可能 100% 准确（Zona 允许运行时动态行为）。

### 路径 B：给 Zona 加最小的类型/签名信息

```
@ fib(1 -- 1) :dup 2 < ? $ :dup 1 - fib :swap 2 - fib + ;
```

`(1 -- 1)` 表示消费 1 个值、产生 1 个值。有了这个，编译器就能用寄存器传参。Forth 社区有这个传统，叫做**栈效应声明**。只需这一行注解，fib 就能生成和 C 一样的代码。

### 路径 C：面向硬件的执行模型设计

不是换汇编生成方式，而是设计一种执行模型，让 Zona 的栈操作天然就是硬件操作。在标准硬件上很难实现，因为 x86/ARM 不是为栈机设计的。但在 FPGA 或自定义 ASIC 上，这是 Chuck Moore 的做法。

## 六、QBE 的关键特性及其在 Zona 中的利用

| QBE 特性 | Zona 中的利用 |
|-----------|---------------|
| 非 SSA 自动修复 | `emit_push`/`emit_pop` 内联展开，不担心临时变量冲突 |
| 不用 phi | 控制流合并点直接 `vsp=0` 清空虚栈 |
| 指针就是 `l` | `dtosi` 直接将 double 转为地址 |
| `copy` 指令 | `T_NUM` 生成 `%t =d copy d_X` |
| `jnz` 接 `w` 类型 | 比较指令结果直接接 `jnz`，省掉中间转换 |
| `alloc4/8/16` | 栈分配，预留给未来结构体构造器 |
| `stored`/`loadd` | 直接操作内存（尝试过用于 Zona 内存，因堆地址间接解析而回退） |

QBE 文档参考：https://c9x.me/compile/doc/il.html

## 八、汇编级实证对比：Zona fib vs C fib

### Zona 编译器生成的 `_zona_fib`（约 100 行汇编）

```asm
_zona_fib:
    stp x29, x30, [sp, -16]!        ; 函数序言
    ; === :dup ===
    ldr w0, [_sp]                    ; 读栈指针
    sub w1, w0, #1                   ; sp - 1
    sxtw x1, w1                      ; 符号扩展
    mul x2, x1, #8                   ; 偏移 = (sp-1) * 8
    ldr d0, [_stack + x2]            ; 从数组读值 ← 内存访问
    ; === 2 < ===
    ldr d1, [常量2]                   ; 加载 2.0
    fcmp d0, d1                      ; 比较
    cset w1, mi                      ; 结果→整数
    scvtf d0, w1                     ; 整数→double     ← 无意义转换
    fcvtzs w1, d0                    ; double→整数     ← 又转回来
    cmp w1, #0                       ; 与0比较
    bne L109                         ; 跳转
    ; === 1 - ===
    ldr d0, [_stack + ...]           ; 再次从数组读 ← 重复内存访问
    fsub d0, d0, 常量1                ; 减1
    str d0, [_stack + ...]           ; 写回数组 ← 内存访问
    add w0, w0, #1
    str w0, [_sp]                    ; 更新sp ← 内存访问
    bl _zona_fib                     ; 递归调用
    ; ... 重复的 load/store/sp 操作 ...
```

关键问题：每次栈操作都涉及**全局数组读写**（`_stack` + `_sp`），比较操作有**双重类型转换**（整数↔double）。

### C 编译器生成的 `_fib`（约 20 行汇编）

```asm
_fib:
    fmov d1, #2.0                    ; 常量直接进寄存器
    fcmp d0, d1                      ; 比较（参数 d0 已经在寄存器里）
    b.mi LBB0_2                      ; 直接跳转
    ; n >= 2 分支:
    stp d9, d8, [sp, #-32]!          ; 保存callee-saved寄存器
    stp x29, x30, [sp, #16]
    fmov d1, #-1.0                   ; 常量直接进寄存器
    fadd d1, d0, d1                  ; n-1，d0就是n，已在寄存器
    mov.16b v9, v0                   ; 保存n到v9（寄存器！）
    mov.16b v0, v1                   ; 参数进d0
    bl _fib                          ; 递归调用
    mov.16b v8, v0                   ; 保存结果到v8（寄存器！）
    fmov d0, #-2.0
    fadd d0, d9, d0                  ; n-2，从v9读n（寄存器！）
    bl _fib
    fadd d0, v8, d0                  ; 相加，全在寄存器
    ldp d9, d8, [sp], #32
LBB0_2:
    ret
```

关键差异：**零内存访问** — 参数通过 d0 寄存器传入，中间值保存在 v8/v9 寄存器，比较直接跳转。

### 量化对比

| 指标 | Zona _zona_fib | C _fib |
|------|---------------|--------|
| 汇编行数 | ~100 | ~20 |
| 内存读/写 | 6-8 次/递归调用 | 0 次（热路径） |
| 类型转换 | 4 次/比较操作 | 0 次 |
| 寄存器传参 | 否（全局数组） | 是（d0） |
| 寄存器返回 | 否（全局数组） | 是（d0） |
| callee-saved 保存 | 无 | 有（d8-d9） |

**5:1 的代码量差距，核心来自全局数组 vs 寄存器传参。**

## 九、结论

Forth 的"直接掌控硬件"精神，在现代编译器语境下的等价物是：**编译器理解数据流，让值留在寄存器里，而不是在内存中搬来搬去。** 这是一种编译策略，不是代码生成方式的选择。

当前 Zona 虚栈优化已经在做"面向硬件"的事 — 把栈操作映射到 QBE 临时变量（→ 寄存器）。只是函数调用边界没法跨过去，因为缺少栈效应信息。如果能补上这最后一块，QBE 就足够了，不需要绕过它。

性能瓶颈不在编译器实现，也不在后端选择，而在**语言语义本身** — 无类型、无参数、全局栈的设计让编译器不得不生成大量内存操作代码。缩小与 C/Hare 的差距，本质上是给编译器更多语义信息的问题。

## 附录：关键知识参考

### 高性能 Forth 技术概览

| 技术 | 描述 | 性能影响 |
|------|------|----------|
| TOS 寄存器缓存 | 栈顶元素始终保存在硬件寄存器中 | 消除大部分栈内存访问 |
| 双 TOS 缓存 | 栈顶两个元素在寄存器中 | 对 `:swap`、`+` 等双操作数指令尤其有效 |
| 子程序线程 | 每个词编译为 call/ret 序列 | 比间接线程更好的分支预测 |
| 内联原语 | 底层操作直接插入机器码 | 消除 call/ret 开销 |
| 超级指令 | 高频组合合成单条机器码 | 减少指令数 |
| 栈效应声明 | 字签名声明消费/产生数量 | 使寄存器传参成为可能 |

# Zona V2 解释器设计

> 本文档探讨 V2 解释器的技术方案，包括字节码 VM、直接线程、分离栈、基线 JIT 等方案，以及解释器与编译器的协同策略。

## 一、为什么 Node.js 那么快

Node.js 不是解释器，它有五层执行引擎：

```
JS 源码
  ↓ 解析
字节码（Ignition 解释器执行）
  ↓ 热点检测（函数被调用多次？）
SparkPlug 基线 JIT（快速编译为机器码，不做优化）
  ↓ 继续热点检测 + 类型反馈收集
TurboFan 优化 JIT（根据运行时观测到的类型做激进优化）
  ↓ 如果类型假设失败
去优化（回退到 Ignition 重新解释）
```

V8 真正快的核心是 **TurboFan 的投机优化** — 观察到 `fib(n)` 的参数 `n` 始终是整数，就生成整数版本的机器码。如果突然传入字符串，就回退。这个"先猜后验"的策略让热路径接近 C。

但 Node.js fib(35) 也要 0.061s（vs C 0.043s），1.4x 的差距 — JIT 没法完全消除动态类型的开销。

## 二、解释器的性能谱系

| 类型 | 代表 | vs C | 原理 |
|------|------|------|------|
| switch 解释器 | CPython, 当前 Zona V1 | 30-50x | while(1) switch(opcode) |
| 直接线程 | Ruby 1.9, Lua 5.0 | 10-20x | computed goto，消除分支预测失败 |
| 字节码 + 类型标签 | LuaJIT 解释器 | 5-10x | 紧凑字节码 + tagged value 快速分派 |
| 基线 JIT | V8 SparkPlug | 2-5x | 第一次编译就生成机器码 |
| 追踪 JIT | LuaJIT | 1.5-3x | 记录热路径的执行轨迹，编译为机器码 |
| 方法 JIT | V8 TurboFan | 1.2-2x | 类型反馈 + 投机优化 |

当前 Zona V1 解释器是 switch 解释器，1.63s（38x 于 C）— 性能谱系的最底端。

## 三、V2 解释器的关键技术

### 3.1 V2 独有优势：栈效应和类型信息

V1 解释器里每个值都是 `double`，每次操作都要检查"这是什么"。V2 有了栈效应声明和类型标记后，解释器可以：

1. **跳过类型检查** — `( d -- d )` 的字，解释器知道参数一定是 double，不需要运行时检查
2. **用 tagged value 快速分派** — 值的类型在标签里，一条位运算就知道分派到哪个处理分支
3. **预计算栈深度** — 有了栈效应，进入/退出字时直接调整栈指针，不需要逐条指令维护

### 3.2 字节码编译

不再解释 token 流，而是编译为紧凑的字节码：

```
源码: :dup 2.0 < ? $ :dup 1.0 - fib :swap 2.0 - fib + ;

字节码:
  DUP          _ 1 字节
  PUSH_D 2.0   _ 1 + 8 字节
  LT_D         _ 1 字节
  JZ_RET       _ 1 字节（条件为真时返回）
  DUP          _ 1 字节
  PUSH_D 1.0   _ 1 + 8 字节
  SUB_D        _ 1 字节
  CALL fib     _ 1 + 4 字节（函数索引）
  SWAP         _ 1 字节
  PUSH_D 2.0   _ 1 + 8 字节
  SUB_D        _ 1 字节
  CALL fib     _ 1 + 4 字节
  ADD_D        _ 1 字节
  RET          _ 1 字节
```

字节码 VM 的优势：
- **紧凑** — 每条指令 1 字节操作码 + 固定长度操作数，CPU 缓存友好
- **快速分派** — 直接线程或间接线程
- **类型化操作** — `ADD_D` vs `ADD_I`，不需要运行时判断类型
- **编译一次，执行多次** — 字被定义时编译为字节码，调用时直接执行
- **JIT 的基础** — 字节码是 JIT 的输入

### 3.3 直接线程分派

C 的 `goto *dispatch[opcode]` — 每个 opcode 的处理以标签开始，执行完后直接跳到下一个 opcode 的标签，不经过中央 switch。

当前 V1 解释器的核心循环：

```c
while (ip < n) {
    switch (toks[ip].type) {
        case T_NUM: push(toks[ip].num); ip++; break;
        case T_PRIM: if (strcmp(toks[ip].text, ":dup") == 0) { ... } break;
        case T_WORD: call_word(toks[ip].text); break;
    }
}
```

每次循环：switch 判断 → strcmp 匹配原语 → 执行。switch 的分支预测命中率低。

直接线程版：

```c
static void *dispatch[] = {
    &&L_NUM, &&L_PRIM_DUP, &&L_PRIM_DROP, ...
};

while (ip < n) {
    goto *dispatch[code[ip]];
L_NUM:
    push(values[ip]); goto *dispatch[code[++ip]];
L_PRIM_DUP:
    dup(); goto *dispatch[code[++ip]];
L_PRIM_DROP:
    drop(); goto *dispatch[code[++ip]];
}
```

消除 switch 和 strcmp，每次 token 只做一次间接跳转。在 ARM64 上间接跳转约 2-3 个周期（分支预测命中时），switch + strcmp 要 10-20 个周期。

### 3.4 分离栈

V1 统一 double 做不到，V2 有了类型后可以：

```c
int64_t istack[ISTACK_MAX];  _ 整数栈
double  fstack[FSTACK_MAX];  _ 浮点栈
void*   pstack[PSTACK_MAX];  _ 指针栈
int isp, fsp, psp;           _ 各自的栈指针
```

`( d -- d )` 的字只操作 `fstack`，`( i -- i )` 的字只操作 `istack`。不需要标签，不需要类型检查，缓存友好（同类型值连续存储）。

### 3.5 基线 JIT（远期）

利用 macOS 的 `MAP_JIT` + `pthread_jit_write_protect_np()` 在运行时生成和执行 ARM64 机器码。

思路：
1. 字第一次调用时，用字节码 VM 执行
2. 字被调用 N 次后，编译为 ARM64 机器码
3. 后续调用直接执行机器码

有了栈效应声明，基线 JIT 可以直接生成正确的机器码 — 参数从哪个栈/寄存器来，返回值去哪里。不需要 V8 那样的类型反馈和投机优化。

## 四、解释器与编译器的协同

### 字节码是共享的中间表示

V2 解释器的字节码和 V2 编译器的 ARM64 代码生成，概念上是一样的 — 把 Zona 的语义操作映射到更低级的指令序列。

字节码指令和 ARM64 指令可以是一对一映射：

| 字节码 | ARM64 |
|--------|-------|
| `PUSH_D val` | `fmov d0, #val` |
| `ADD_D` | `fadd d0, d0, d1` |
| `SUB_D` | `fsub d0, d0, d1` |
| `LT_D` | `fcmp d0, d1; b.mi` |
| `CALL idx` | `bl $func_name` |
| `DUP` | `mov.16b v1, v0` |
| `RET` | `ret` |

编译器本质上是"字节码到 ARM64 的翻译器"，实现复杂度比从零写一个代码生成器低得多。

### 编译管线

```
                zona 源码
                    ↓
            tokenizer + 栈效应验证
                    ↓
              字节码编译器
                    ↓
                字节码（IR）
               ↙        ↘
    解释器执行          编译器翻译
    （开发/调试）       （部署/性能）
        ↓                  ↓
    字节码 VM          ARM64 汇编
                          ↓
                      cc 链接
                          ↓
                      原生可执行
```

字节码是核心的中间表示。解释器和编译器共享前端，分叉在后端。

### 使用场景

| 场景 | 用解释器 | 用编译器 |
|------|---------|---------|
| 开发调试 | 即时反馈、`:trace` 追踪 | 编译等待、难以调试 |
| REPL | 天然支持 | 需要实时编译 |
| 小脚本 | 启动快 | 编译开销 > 运行时间 |
| 计算密集 | 慢 | 快 |
| 生产部署 | 不推荐 | 推荐 |

## 五、分阶段实现

### 阶段 1：字节码 VM + 直接线程 + 分离栈

预期 vs C：5-10x（从 V1 的 38x 缩小）
开发量：中等
基础设施：字节码编译器 + VM 循环 + 分离栈

### 阶段 2：ARM64 编译器（字节码 → 汇编翻译器）

预期 vs C：1.0-1.3x
开发量：中等（因为字节码已经定义好了指令映射）
基础设施：ARM64 代码选择器 + 汇编输出

### 阶段 3：基线 JIT（如果需要）

预期 vs C：1.5-3x
开发量：大
依赖：MAP_JIT + ARM64 代码生成（与编译器共享后端）

## 六、开放问题

### 6.1 字节码指令集设计

需要定义完整的字节码指令集 — 操作码编码、操作数格式、调用约定。这是解释器和编译器共享的契约，需要仔细设计。

初步设计方向：
- 1 字节操作码，最多 256 条指令
- 操作数紧跟操作码，长度由操作码决定
- 类型化操作码（ADD_I / ADD_D / ADD_L）
- 跳转使用相对偏移（16 位有符号整数）

### 6.2 分离栈与统一栈的选择

分离栈性能更优，但增加了：
- 栈操作原语的复杂度（`:swap` 是交换同类型栈，还是跨类型？）
- 调试体验（三个栈 vs 一个栈）
- 与 C 互操作的接口复杂度

初步倾向分离栈，但需要原型验证。

### 6.3 REPL 与增量编译

字节码 VM 天然支持 REPL — 输入一行，编译为字节码，立即执行。但需要处理：
- 新定义的字如何加入运行时字典
- 修改已有字的定义如何更新字节码
- 模块导入的增量处理

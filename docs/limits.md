# 资源上限

Zona 使用静态数组存储运行时数据，所有上限为编译期 `#define` 常量，集中定义在 `src/zona.h`。

## 设计意图

- 避免动态分配带来的内存管理复杂度
- 上限值为保守默认，可按需调大重新编译
- 部分上限超限有报错，部分静默溢出——后者是已知隐患，待统一处理

## 当前限制

| 常量 | 值 | 作用域 | 说明 |
|------|-----|------|------|
| `DICT_MAX` | 1024 | 全局 | 字典条目数，一个 `@ ... ;` 占一个槽位 |
| `TOK_MAX` | 4096 | 单文件 | 一个 `.zona` 文件的 token 上限 |
| `WORD_BODY_MAX` | 256 | 单个字 | 一个字定义体的 token 上限 |
| `STACK_MAX` | 256 | 全局 | 数据栈深度 |
| `RSTACK_MAX` | 256 | 全局 | 调用栈深度 |
| `MEM_CELLS` | 65536 | 全局 | 静态内存单元数（~512 KB） |
| `HEAP_MAX` | 1024 | 全局 | 动态分配（`:alloc`）次数上限 |
| `USE_MAX` | 64 | 全局 | `:use` 引入文件数上限 |
| `FHANDLE_MAX` | 16 | 全局 | 同时打开文件数上限 |

## 各限制的实际影响

### DICT_MAX — 最敏感的瓶颈

每个 Zona 字（`@ name ... ;`）占用一个字典槽位。标准库约占用 15 个槽位。1024 个槽位约对应 4000-8000 行 Zona 代码。

之前的值为 256，约对应 1000-1500 行代码。提升到 1024 可支撑中小型程序。

### TOK_MAX + USE_MAX — 通过拆文件扩容

单文件 4096 token 约等于 400-800 行代码。配合 `:use` 可拆分为最多 64 个文件，理论上总代码量可达 ~25000 行。

### WORD_BODY_MAX — 几乎不可能触发

单个字定义体 256 token 的上限，正常编写无法达到。主要是防止无限循环导致的内存问题。

### STACK_MAX / RSTACK_MAX — 够用

数据栈深度 256 对于 Zona 的后缀表达式和控制流已经足够。调用栈 256 层也远超实际需求。

### MEM_CELLS — 字符串存储

每个 cell 存一个 `double`（8 字节），65536 cell ≈ 512 KB。字符串按字符拆分存储（每个字符占一个 cell），效率不高。对于文本密集的程序可能成为瓶颈，后续可优化为字节级存储。

### 溢出处理的不一致

当前部分上限超限时打印错误并继续执行，部分直接 `exit(1)`，部分未做边界检查可能静默溢出。这是已知隐患，统一超限处理是待做事项。

## 调整指南

所有上限在 `src/zona.h` 中修改，重新编译即可：

```
cc src/zona.c -o zona -lm -lreadline
cc src/zonac.c -o zonac -lm
```

内存开销估算：
- 每个 `Word` 约 71 KB（含 256 token 的 body 数组）
- `dict[1024]` 约 72 MB（BSS 段，不占磁盘）
- 每个 `Token` 约 276 字节（含 256 字节的 text 缓冲区）

当前设计以简单优先，token 内部的固定大小缓冲区较为浪费。后续可优化为动态分配，但目前规模下不构成实际问题。

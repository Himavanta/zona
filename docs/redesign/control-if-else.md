# 在虚栈追踪的情况下控制流会有限制，当前的情况是怎么回事，这个问题是怎么回事

好问题，让我把这个问题讲透。

## 根本矛盾：虚栈是编译期概念，控制流是运行期分叉

**虚栈**（`vstack[]` + `vtype[]`）是编译器的**内部记事本**——编译 `@ abs l:l :dup 0 < ? neg ;` 时，编译器逐 token 记录"当前哪个 QBE temp 在栈上，是什么类型"。生成完 IR，记事本就扔了。

**控制流（`?`）** 在运行时制造了**两条不同的路径**，每条路径对栈的操作可能不同。但生成 IR 时编译器必须一次性把两条路径的代码都写好。

## 具体问题：两条路产出不同数量的值

看这个例子：
```
42 . 1 ? 99 .
```

处理 `?` 时的状态：

| 步骤 | vstack |
|------|--------|
| 执行 `42` 后 | `[42]` |
| 执行 `.` 后 | `[]` |
| 执行 `1` 后 | `[1]` |
| 遇到 `?`，pop 条件 | `[]` ← **这里是 save 点** |
| **真分支** 执行 `99` | `[99]` → vsync 写入运行时栈 `[99]` |
| **假分支** 什么都不做 | `[]` → vsync 写入运行时栈 `[]` |
| 合并点 | 应该 pop 几个值？ |

**问题本质**：真分支写入了 1 个值（99），假分支写入了 0 个值。合并时不知道 pop 几个。

当前代码的处理方式是：
```c
if (!has_else) {
    out_vsp = save_vsp;   // ← 用真分支前的 vsp 覆盖真分支的结果！
}
```

**这是个 bug**——真分支明明产出了 1 个值，但 `out_vsp` 被覆盖成了 0。所以合并点 pop 0 次，99 留在运行时栈里被后续操作当垃圾读到。

## 核心限制：虚栈方案要求分支产出数必须一致

用 vsync/reload 方案做控制流，有一个**硬性约束**：

> `?` 的两个分支必须产出**相同数量**的值，否则运行时栈状态无法对齐。

| 情况 | 真分支产出 | 假分支产出 | 能否对齐 |
|------|-----------|-----------|---------|
| `? 99` | 1 | 0（无 `!`） | ❌ 需要推哑元 |
| `? 99 ! 200` | 1 | 1 | ✅ |
| `? :drop` | -1 | 0（无 `!`） | ❌ 需要弹 dummy |
| `? :drop ! :drop` | -1 | -1 | ✅ |

## 当前代码的具体 bug

当前 `gen_word` 中的 `?` 处理器有一个明确的 bug：

```c

jmp @l1

/* 假分支（无 !） */
else {
    out_vsp = save_vsp;  // ← 覆盖成 0！真分支的信息丢了
}
```

合并点：
```c

for (int i = 0; i < out_vsp; i++)   // out_vsp=0 → pop 0 次！
    emit_pop_typed(...);
```

99 被推入了运行时栈但永远没人 pop，后续操作读到脏数据。

## 修复方向

有两种思路：

### 思路 A：要求分支平衡（类型安全）

编译器在编译时报错：`?` 不带 `!` 时，真分支的净栈变化必须为 0。这样假分支（什么都不做）自然平衡。

优点：编译期捕获错误，无运行时开销。
缺点：限制表达能力——`cond ? 42` 这种常见写法需要改写成 `cond ? 42 ! 0`。

### 思路 B：假分支自动推哑元（当前意图）

编译真分支时记录净产出（`out_vsp - save_vsp`），假分支自动推对应数量的 0。这就是当前代码想做的但 `out_vsp` 被错误覆盖了。

优点：`?` 保持灵活，任何单 token 都可用。
缺点：假分支产出无意义的 0，可能掩盖逻辑错误。

### 思路 C：用 phi 节点（彻底解决）

两条分支产出不同数量时不能用简单的 vsync/reload，但用 **phi 节点** 可以——phi 根据执行路径选值，天生处理分叉。

不过 phi 的复杂度在于：如果两个分支产出值**数量不同**，再多的 phi 也无法对齐。所以 phi 方案下仍然要求分支产出数一致。

---

**本质上**：这不是 vsync/reload 的锅，而是**栈语言的静态栈效应要求控制流的动态栈效应必须对齐**——无论用什么后端，这个约束都存在。V2 的栈效应签名已经提供了对齐检查所需的信息，只是编译器当前没有实施这个检查。

# 性能优化计划

## 目标

编译器 fib(35) 从 0.48s → 接近 C 的 0.027s（目标 5-10x 提升）。

## 核心思路

编译时虚拟栈：栈上的值直接用 QBE 临时变量（`%t`），不经过 push/pop 函数调用。只在控制流边界同步到运行时栈。

渐进式——知道怎么做就优化，不知道的退回到保守路径。不改变语言，不增加运行时复杂度。

---

## 阶段一：虚拟栈骨架 ✅

**已完成** (2026-04-30).

- T_NUM：生成 `%t =d copy d_X`，vpush 记录
- 算术符号（+ - * / % ^ > < =）：vpopr 获取操作数，直接在临时变量上运算，vpush 结果
- T_STR、T_PRIM、T_WORD、`.`、`&`、`#`、`?`、`$`、`~`：先 vsync() 再走原路径
- `?` 分支结束处 vsync，合并点虚栈清空
- 函数体结束（`;`）前 vsync
- `vpopr()`：虚栈空时从运行时栈 reload，实现虚实自动切换

**成果**: fib(35) 0.48s → 0.29s（1.65x），loop(10M) 0.08s → 0.07s（1.14x）。

**已知遗漏**（阶段一该做但没做的）：

- `:dup` `:swap` `:drop` `:over` `:rot` 没有虚栈化——它们在虚栈上只是编译器内部记账（交换 temp ID），零运行时开销，改动 20 行
- `:type` 虚栈化——字符串 addr/len 在虚栈里，不需要 sync 两次
- `&` `#` 虚栈化——`:here` + 算术算地址在虚栈，最后 sync 一次
- `?` 不需要全部 vsync——如果虚栈只有一个值（条件），其他值在分支里不动，不该落地

## 阶段二：小字内联 ⏳

body ≤ N token 的字不生成独立函数，直接在调用点展开。被内联字的 token 继承调用方的虚栈，不触发 vsync，无函数调用开销。

- 标准库字几乎都可内联：`neg`(3)、`sq`(3)、`abs`(5)、`min`(6)、`max`(6)、`not`(3)、`nip`(3)
- 递归字不能内联——调用时判断"调用的字名 ≠ 当前正在编译的字名"
- 预期额外 2-3x 提升（非递归代码）

## 阶段三：调用约定优化 ⏳

**问题**: 当前任何字调用都 vsync() 全部清空，即使调用只需要栈顶几个值。

**3a. 栈效应分析**

静态分析每个字的栈效应（消耗 N 个值，产出 M 个值）。调用时只 vsync 顶部 N 个值作为参数，虚栈其余值保留不动。调用后从运行时栈 reload M 个返回值到虚栈顶。

**3b. 尾调用返回优化**

如果字的最后一个操作为 `;` 且虚栈只有一个值，约定将其放在已知 temp 中，调用方直接读取——省去 push + pop 两个来回。

**3c. 叶字优化**

不调用其他字、不使用控制流的"叶字"，内部完全不 sync，只在最后 ret 前 vsync。消除叶字内全部 push/pop。

## 阶段四：常量折叠 ⏳

编译时计算常量表达式（`3 5 +` → `8`）。虚拟栈值均为常量时直接计算替换。收益小但几乎零成本。

## 激进方向（远期）

### 打倒运行时栈

如果栈效应分析覆盖到全程序，`$sp`、`$stack`、`$zona_push/pop` 全套运行时可以退化——只在间接调用和未来高阶函数时保留。把 Forth 的栈机语义在编译期翻译成寄存器机。

### 类型特化

栈上全是 double，但很多值实际是整数（循环计数、地址、布尔）。编译器追踪"这个值看起来像整数吗？"，生成两套代码：

```
快速路径: 整数加法（QBE add w）→ 检查不溢出 → 跳过 push/pop
慢速路径: 退回 double 加法 + push/pop
```

fib(35) 全程整数，快速路径 100% 命中。

### 生成 C 而非 QBE

每个字 → `static double fn(double* stack, int* sp)`，cc/clang 替你做寄存器分配、内联、循环展开。作为 release 模式可选。与"打倒运行时栈"同方向，不同实现路径。

### 解释器首字符跳转

`exec_prim` 的 `strcmp` 链（21 个线性匹配）改成首字符跳转表：

```c
switch (name[1]) {
    case 'd': // :dup, :drop
    case 's': // :swap, :stack
    case 'o': // :over
    ...
}
```

省分支预测失败，预期 1.5-2x 解释器提升。

## fib 的天花板

fib(35) 约 2930 万次函数调用。即使所有内部操作虚栈化，递归调用自身的开销仍无法消除——每次调用仍有函数序言/尾声开销。

对于 fib，达到 C 的 2-3x（约 0.06-0.08s）是合理预期。5-10x 应该看非递归算术密集的 benchmark。

## 廉价优化（改动小，见效快）

### 运行时函数内联展开

`$zona_push` 只有 7 条 QBE 指令。不在调用点 `call $zona_push`，而是直接展开这 7 条指令。没有 call/ret 开销，QBE 自己能发现相邻的 `store/load` 冗余对并消除。

**意义**：以完全不同的路径实现了虚栈的效果——不跟踪 temp ID，让后端优化器替你消除冗余。与虚栈互补：虚栈是"编译器知道就不生成"，内联是"生成了让 QBE 自己清"。

### 栈顶缓存（解释器）

不把栈全放数组，栈顶 3-5 个值放 C 局部变量（硬件寄存器）。大部分 Forth 操作只用栈顶 1-2 个值，把数组读写变成寄存器 copy。

```c
// 当前: 每次 push/pop 都读写 stack[sp]
// 优化后: 保持 top, second, third 在局部变量，只有溢出才写回数组
```

Gforth 和 SwiftForth 的看家本领。

### 超指令

识别常见 token 序列，编译时模式匹配合为一条高效指令：

| 序列 | 替换为 | 说明 |
|------|--------|------|
| `:dup +` | add-with-dup | 极常见 |
| `1 - ~` | decrement-and-loop | 循环惯用模式 |
| `:swap :drop` | nip | 标准库已有 |
| `:dup 0 = ? $` | exit-if-zero | 循环出口惯用 |

本质是编译器层面的 peephole，但针对 Zona 的特定惯用模式，不是通用优化。

### 字节级字符串存储

当前每个字符占一个 cell（8 字节）。改成每 cell 打包 8 个字节：

```
'hello world' → 11 cell (88 bytes) → 2 cell (16 bytes)
```

`:type` 时按字节偏移解包。不影响语义，只改存储布局和 `:type` 的实现。

# 性能优化计划

## 目标

编译器 fib(35) 从 0.48s → 接近 C 的 0.027s（目标 5-10x 提升）。

## 核心思路

编译时虚拟栈：栈上的值直接用 QBE 临时变量（`%t`），不经过 push/pop 函数调用。只在控制流边界同步到运行时栈。

## 阶段一：虚拟栈骨架 ✅

**已完成** (2026-04-30).

- T_NUM：生成 `%t =d copy d_X`，vpush 记录
- 算术符号（+ - * / % ^ > < =）：vpopr 获取操作数（虚栈空时从运行时栈 reload），直接在临时变量上运算，vpush 结果
- T_STR、T_PRIM、T_WORD、`.`、`&`、`#`、`?`、`$`、`~`：先 vsync() 再走原路径
- `?` 分支结束处 vsync，合并点虚栈清空
- 函数体结束（`;`）前 vsync

**成果**: fib(35) 0.48s → 0.29s（1.65x），loop(10M) 0.08s → 0.07s（1.14x）。

纯算术代码收益远超此数（`time` 测量到 0.00s user）。fib 提升有限因为每次递归调用触发 vsync，虚栈被反复清空，函数调用开销仍占主导。

## 阶段二：小字内联 ⏳

### 原理

body ≤ N token 的字不生成独立函数，直接在调用点展开：

```
原: @ neg 0 :swap - ;
    ... neg ...

内联后: ... 0 :swap - ...   (三个 token 直接插入)
```

- 被内联字的 token 继承调用方的虚拟栈，不触发 vsync
- 没有函数调用开销（call/ret、保存/恢复寄存器）
- 标准库字几乎都可内联：`neg`(3)、`sq`(3)、`abs`(5)、`min`(6)、`max`(6)、`not`(3)、`nip`(3)、`nl`(3)

### 非递归约束

递归字（如 `fib`）不能内联——展开会导致无限循环。只在调用点判断：调用的字名 ≠ 当前正在编译的字名时才内联。

### 预期收益

非递归代码中消除短字调用开销，配合虚拟栈使更多代码成为"直线区域"。fib 中 `fib` 自身不能内联，但标准库辅助字可以。

## 阶段三：调用约定优化 ⏳

### 问题

当前任何字调用都走：

```
vsync()        → 把虚栈所有值推到运行时栈
call $word     → 字从运行时栈 pop/push
vpopr() 逐个   → 调用方从运行时栈逐个 reload
```

`vsync()` 是"全部清空"，即使调用只需要栈顶几个值，虚栈中无关的值也被迫落地。

### 方案：惰性 vsync + 调用后 reload

**3a. 尾调用返回优化**

如果字的最后一个操作为 `;` 且虚栈只有一个值，不 push 到运行时栈，而是约定将其放在某个已知 temp 中。调用方直接从该 temp 获取返回值——省去 push + pop 两个来回。

**3b. 惰性参数传递**

逐字分析栈效应（消耗 N 个值，产生 M 个值）。调用时只 vsync 顶部 N 个值作为参数，虚栈其余值保留不动。调用后从运行时栈 reload M 个返回值到虚栈顶。

需要一次简单的静态分析扫描，对每个字记录 `(in, out)` 计数。

### 预期收益

额外 2-3x，叠加阶段二可达 5-8x。但递归调用（fib 自身）仍无法绕过——这是 Zona 调用模型的本质约束。

## 阶段四：常量折叠 ⏳

- 编译时计算常量表达式：`3 5 +` → `8`
- 虚拟栈值均为常量时直接计算替换
- 收益小但几乎零成本实现

## fib 的天花板

fib 是递归密集型测试，瓶颈是 ~2930 万次函数调用。当前每次调用约有 8 次 C 函数调用开销（push/pop for args+return+sync）。即使用完阶段二/三，只要递归调用自身还在，就永远无法接近 C 的 0.027s（C 的每次调用只有 3-4 条指令）。

**对于 fib 测试，达到 C 的 2-3x（约 0.06-0.08s）是合理预期。** 要 5-10x 应该看非递归算术密集的 benchmark。

## 暂不做

- 尾调用优化（`~` 已是循环）
- peephole（虚拟栈自然消除 push/pop 对）
- 解释器优化（收益太小）
- QBE 替换（生成代码质量够好）

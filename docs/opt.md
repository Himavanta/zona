# 性能优化计划

## 目标

编译器 fib(35) 从 0.48s → 接近 C 的 0.027s（目标 5-10x 提升）。

## 核心思路

编译时虚拟栈：栈上的值直接用 QBE 临时变量（`%t`），不经过 push/pop 函数调用。只在控制流边界同步到运行时栈。

## 阶段一：虚拟栈骨架 ✅

**已完成** (2026-04-30).

- T_NUM：生成 `%t =d copy d_X`，vpush 记录
- 算术符号（+ - * / % ^ > < =）：vpopr 获取操作数（虚拟栈空时从运行时栈 reload），直接在临时变量上运算，vpush 结果
- T_STR、T_PRIM、T_WORD、`.`、`&`、`#`、`?`、`$`、`~`：先 vsync() 再走原路径
- `?` 分支结束处 vsync，合并点虚栈清空
- 函数体结束（`;`）前 vsync

**成果**: fib(35) 0.48s → 0.29s（1.65x），loop(10M) 0.08s → 0.07s（1.14x）。Zona 编译器性能从 Ruby 级提升至 Ruby 与 Node.js 之间。

**局限**: fib 提升有限因递归调用的函数开销仍占主导。直线算术已大幅优化，但字调用点仍有 push/pop 开销。

## 阶段二：小字内联 ⏳

- body ≤ 5 token 的字直接内联到调用点
- 配合虚拟栈扩大"直线代码"区域
- fib 的递归调用不能内联，但 `neg`、`sq`、`abs` 等标准库字可以
- 预期额外 2-3x 提升（非递归代码）

## 阶段三：常量折叠 ⏳

- 编译时计算常量表达式
- 虚拟栈值均为常量时直接计算替换
- 收益小但几乎零成本实现

## 暂不做

- 尾调用优化（`~` 已是循环）
- peephole（虚拟栈自然消除 push/pop 对）
- 解释器优化（收益太小）
- QBE 替换（生成代码质量够好）

# 性能优化计划

## 已做

### 阶段一：虚栈骨架 ✅

T_NUM、算术符号走虚栈；控制流边界 vsync。fib(35) 0.48s → 0.29s。

---

## 下一步（按优先级）

### 阶段二：补齐虚栈遗漏 ⏳

阶段一改了算术符号和 T_NUM。下面这些同样可以虚栈化、改动极小，属于"该做但漏了"：

| 操作 | 当前 | 虚栈后 | 改动 |
|------|------|--------|------|
| `:dup` | call peek + call push | `vpush(vpeek())` | 1 行 |
| `:drop` | call pop | `vpopr()` 丢弃 | 1 行 |
| `:swap` | 两次 pop + 两次 push | 交换虚栈顶两个 temp ID | 3 行 |
| `:over` | 两次 pop + 三次 push | vpush(虚栈倒数第二个 temp) | 2 行 |
| `:rot` | 三次 pop + 三次 push | 重排虚栈顶三个 temp ID | 4 行 |

**收益**: 栈密集型代码零运行时开销，改动 ~20 行。

### 阶段三：运行时函数内联展开 ⏳

不再生成 `call $zona_push`，直接在调用点展开 push 的 7 条 QBE 指令。所有运行时函数（push/pop/peek/print/mem_load/mem_store）都内联。

**为什么这比手工 peephole 好**：展开后 QBE 自己会发现相邻的 store→load 冗余对并消除。我们不用写"检测 push 后紧跟 pop"的逻辑——QBE 替你做。

**额外收益**：配合阶段二的虚栈化，只剩控制流边界的 push/pop 会生成展开代码。两者叠加，push/pop 几乎完全消失。

**注意**：这会让 .ssa 文件变大（原来一个 call 变成 7 条指令），但最终编译产物不受影响——QBE 会把冗余的消除掉。

### 阶段四：超指令 ⏳

识别 Zona 惯用 token 序列，编译时替换为高效版本：

| 序列 | 替换 | 场景 |
|------|------|------|
| `:dup +` | dup-add | `:dup n +` 极常见 |
| `1 - ~` | dec-loop | 倒计数循环 |
| `:dup 0 = ? $` | exit-if-zero | 循环出口 |
| `:swap :drop` | nip | 标准库已有此字 |
| `:dup 0 < ? neg` | abs | 就是标准库 abs 的实现 |

**为什么放在阶段三之后**：运行时函数内联展开后，push/pop 调用已消除，超指令进一步把"多条 QBE 指令"换成"单条等效指令"，让 QBE 的寄存器分配更高效。

### 阶段五：小字内联 ⏳

body ≤ 8 token、非递归的字直接在调用点展开。标准库字几乎全在内联范围内。

**为什么放在后面**：前面阶段已经把调用开销降到很低。内联的增量收益变小了——但仍然重要，因为内联扩大"直线代码区域"，让虚栈和超指令能连续工作更长距离。

## 更远的方向

### 解释器：栈顶缓存 + 首字符跳转

- 栈顶 3 个值放 C 局部变量（硬件寄存器），大部分操作变成寄存器 copy
- `exec_prim` 的 strcmp 链改成首字符 switch，省分支预测失败

### 调用约定优化

静态分析每个字的栈效应 `(in → out)`，调用时只同步参数，不同步无关值。叶字（无内部调用的字）内部完全不 sync。

### 类型特化

追踪值的实际类型，整数走 QBE `w` 指令，浮点走 `d` 指令。fib 全程整数，100% 命中快速路径。

### 生成 C（release 模式）

替代方案：生成 C 代码而非 QBE IR，让 cc/clang 做寄存器分配和内联。开发时用 QBE（快），发布时走 C（最优）。

---

## 存储优化

**字节级字符串存储**: 当前每个字符占一个 cell（8 字节），`'hello world'` = 88 字节。改成每 cell 打包 8 个字符 = 16 字节。`:type` 按字节偏移解包，不影响语义。

---

## fib 天花板

递归调用自身无法内联，每次调用仍有 QBE 函数序言/尾声开销。预计 fib(35) 最终可达 0.06-0.08s（C 的 2-3x）。更接近 C 需要类型特化或生成 C。

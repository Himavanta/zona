# 模块系统设计讨论

> 当前 Zona 使用最简单的扁平 `:use` 方案。本文档记录命名模块系统的设计探索和遗留问题，供未来参考。

## 当前方案：扁平引入

使用 `:use` 原语引入其他文件，所有定义摊平到当前全局字典。

```
:use './math.zona'
:use '../lib/utils.zona'
```

规则：
- 路径必须以 `./` 或 `../` 开头
- 同一文件只会被引入一次（避免循环引入）
- 路径相对于当前文件所在目录
- `:use` 必须行首、独占一行、不在 `@ ... ;` 内部
- 字可覆盖重定义，后定义遮盖先定义

无前缀路径保留给未来可能的标准库或包管理使用。

## 命名模块方案（未实现）

### 基础语法

`:use` 后跟模块别名和路径，创建独立命名空间：

```
:use math '../math.zona'
:use net  '../net/http.zona'
```

模块名遵守字的命名规则。模块名存储在独立的模块表中，不与用户字字典冲突。

### 成员访问

`.` 作为成员访问分隔符，与结构体字段访问统一：

```
math.abs .          _ 从 math 模块取 abs
net.get .           _ 从 net 模块取 get
```

`字.字` 在词法层合并为单一 token。空格规则：`.` 有空格 = 打印，无空格 = 成员访问。

### Barrel 导出

一个模块汇集多个子模块的扁平引入，对外提供统一入口：

```
_ all.zona
:use './math.zona'
:use './io.zona'
:use './logic.zona'
```

使用者以命名模块方式引入：

```
:use all '../std/all.zona'
all.abs .
all.nl .
```

### 传递性

模块引入关系**不传递**。a 引入 b，b 引入 c，a 无法访问 c 的字。a 需要 c 时应直接引入，去重机制保证 c 只加载一次。

## 已放弃的方案

### `"` 作为模块分隔符

放弃原因：`'` 和 `"` 同屏视觉混淆，且无法与 struct 字段访问统一（struct 字段也用 `.`）。

### `,` 替代 `.` 做 print

放弃原因：`. ` 打印保持 Forth 传统，与成员访问通过空格规则共存——词法层判断，不需要运行时上下文。

### `:reexport` 原语

不需要——无别名 `:use` 在模块内部摊平即实现 barrel export。

### 选择性导入

`:use math '../math.zona' abs sq min max` 只导入指定字。放弃原因：增加了 `:use` 行格式的复杂度，且模块内部依赖的闭包解析增加了实现难度。

## 核心设计决策

- **模块名和字名分开存储**（模块表 vs `dict[]`）：`@ math` 定义同名字不覆盖模块名，因为 `math` 和 `math.abs` 通过 `.` 有无后缀自然区分
- **`.` 不做 `,` 替代**：打印保持 Forth 传统的 `.`，与成员访问通过空格规则共存
- **`[ ]` `( )` `{ }`** 是最有价值的待选符号，预留给错误处理等关键功能

## 遗留问题

### 模块嵌套与可见性

如果 a 用命名方式引入 b，c 再引入 a，c 能否通过 `a.b.abs` 访问 b 的字？

- **允许** → 链路自然穿透，不需要 barrel 文件，但内部依赖自动变成公开 API
- **不允许** → 需要区分"导入为内部依赖"和"导入为公开导出"，两种 `:use` 模式共存

没有完美答案。当前选择搁置此问题，沿用最简单的扁平方案。

### 调用链长度

允许 `a.b.c.d.fn` 会导致调用非常长。Forth 传统强调短小精悍，深层命名空间与此有张力。

### 文件夹即模块

`:use './math'` 引入目录下所有 `.zona` 文件。暂不考虑，`:use` 基础上加目录遍历即可，无新设计问题。

## 结论

当前扁平 `:use` 方案简单可行，覆盖了所有实际需求。命名模块系统在以下问题想清楚之前，不急于实现：

1. 模块可见性模型（私有依赖 vs 公开导出）
2. 嵌套深度与调用简洁性的平衡
3. 是否需要区别于 `:use` 的新原语来表达命名引入
